### 1. Module 1, Lesson 1: Introduction to Data Parallel Architectures

#### 1.1 Course Framework and Objectives

The primary goal of this curriculum is to enable the reader to describe GPU architecture functionality, optimize GPU programs, and conduct advanced research in compilers or hardware accelerators.

- **Prerequisites:** Proficiency in C++/Python and a fundamental understanding of the classic 5-stage CPU pipeline are required. Prior GPU-specific programming experience is not a prerequisite.
- **Core Competencies:** Understanding the evolution of hardware accelerators and establishing a foundation for CUDA features.

#### 1.2 The Nature of the GPU

While originally developed for 3D graphics, the **Graphics Processing Unit (GPU)** has evolved into a versatile parallel processor essential for High-Performance Computing (HPC) and Machine Learning (ML) acceleration.

**CPU vs. GPU: A Comparative Technical Analysis**:

- **Target Applications:** CPUs are engineered for **latency-sensitive** applications, focusing on the maximum speed of single-threaded execution. GPUs are engineered for **throughput-sensitive** applications, designed to handle thousands of simultaneous threads.
- **Exception Handling:** CPUs prioritize **precise exceptions**, which are vital for debugging and program correctness. GPUs do not prioritize precise exceptions, focusing instead on computational throughput.
- **System Role:** The CPU acts as the **Host**, managing the Operating System (O/S) and Input/Output (I/O) operations. The GPU acts as an **Accelerator**, offloading parallel compute tasks from the host.
- **Instruction Set Architecture (ISA):** CPUs typically utilize **Open/Public ISAs** to ensure broad software compatibility. GPUs often use **Closed ISAs**, where a driver translates code between different hardware-specific ISAs.
    - **PTX (Parallel Thread Execution):** A virtual ISA introduced to provide a public, stable version of the GPU ISA.

#### 1.3 Execution and Programming Models

The fundamental distinction in how processors handle instructions and data is categorized through Flynn’s concepts:

- **SISD (Single Instruction, Single Data):** The traditional model for single-threaded CPU operations.
- **SIMD (Single Instruction, Multiple Data):** A model where a single instruction is applied across multiple data elements simultaneously.
    - _Analogy:_ Multiple chefs performing the exact same chopping motion on different vegetables.
    - _Strengths:_ Highly efficient for data-parallel tasks with consistent structures (e.g., image processing).
    - _Limitations:_ Struggles with branching or conditional logic due to "lockstep" execution requirements.
- **SPMD (Single Program, Multiple Data):** The primary programming model for GPUs, emphasizing parallelism across a single program.
    - _Analogy:_ A team of artists working on different sections of a single painting, each using their own techniques on their specific section of data.
    - _Strengths:_ More flexible than SIMD; capable of handling complex workflows, branching, and distributed computing.
    - _Execution Reality:_ While the **programming model** is SPMD, the **hardware execution model** in GPUs is typically **SIMT (Single Instruction, Multiple Thread)**.

### 2. Module 1, Lesson 2: Modern Processor Paradigms

#### 2.1 The Classic CPU Pipeline

The foundation of CPU design is the five-stage pipeline, which facilitates the flow of instructions:

1. **Fetch (Front-End):** Instructions are retrieved from the Instruction Cache (I-cache).
2. **Decode:** The instruction is interpreted. In x86 architectures, a single instruction may be broken down into multiple **micro-operations (uops)**.
3. **Rename/Register Access:** Register files are accessed to prepare operands for the instruction.
4. **Schedule:** The scheduler determines which instructions are ready for execution.
    - _In-order:_ Instructions are selected strictly by program order.
    - _Out-of-order (OoO):_ Instructions are selected as soon as their operands are ready, regardless of original program order, to maximize resource utilization.
5. **Execute:** Actual computation is performed, or memory is accessed.
6. **Write-back:** Results are committed to the register file or memory.

#### 2.2 Techniques for Performance Scaling

- **Superscalar Execution:** These processors fetch, decode, and execute multiple instructions concurrently, effectively aiming for an **Instruction Per Cycle (IPC)** greater than one.
- **Instruction Level Parallelism (ILP):** ILP measures how many instructions can run in parallel.
    - _Independent Instructions:_ Can be executed simultaneously (e.g., two unrelated arithmetic operations).
    - _Dependent Instructions:_ Must be executed sequentially because one relies on the result of the other.
- **Pipeline Depth and Frequency:** Increasing pipeline stages allows for higher clock frequencies but necessitates more accurate **branch prediction** to avoid performance penalties.
- **Cache Hierarchy:** Utilizing larger caches reduces cache misses and significantly decreases memory access latency.
- **Multi-threading:** Enhances performance by switching between multiple threads to find ready instructions, requiring minimal additional hardware resources.
- **Multiprocessors:** Scaling performance by increasing the total number of processor cores, a technique utilized by both CPUs and GPUs.

### 3. Module 1, Lesson 3: How to Write Parallel Programs

#### 3.1 Amdahl’s Law

Amdahl’s Law quantifies the theoretical speedup of an application when using multiple processors.

- **Formula:** $\text{Speedup} = \frac{1}{\frac{P}{N} + S}$
    - **P:** The fraction of the program that can be parallelized.
    - **N:** The number of processors.
    - **S:** The serial section of the program (where $S = 1 - P$).
- **Implication:** Performance scalability is limited by the serial fraction. Even a 10% serial section will eventually impede performance regardless of the number of processors added.

#### 3.2 Parallel Decomposition Strategies

Effective parallel programming requires dividing work efficiently:

- **Task Decomposition:** Different cores perform different types of operations (e.g., Core 1 performs addition while Core 2 performs a Min/Max operation).
- **Data Decomposition:** The data set is split across cores, and each core performs the same operation on its piece of data. This often requires a **reduction operation** (e.g., summing partial results) to achieve the final output.

#### 3.3 Flynn’s Classical Taxonomy Revisited

- **SISD:** Single Instruction, Single Data.
- **SIMD:** Single Instruction, Multiple Data (often used in CPU vector units).
- **MISD:** Multiple Instruction, Single Data.
- **MIMD:** Multiple Instruction, Multiple Data.
- **SIMT (Single Instruction, Multiple Thread):** The specific hardware execution model for GPUs, where one instruction is fetched but operates across many Arithmetic Logic Units (ALUs).

#### 3.4 CUDA Programming Example: Vector Sum

In CUDA, the program is divided into **Host** code (running on the CPU) and **Kernel** code (running on the GPU).

- **Kernel Code Structure:** All cores/threads execute the same function (e.g., `vector_sum()`).
- **Thread Identification:** Each thread uses a unique **Thread ID** to determine which specific data element it should process (e.g., adding a value to its specific index in Array A).
- **Parallelism:** All threads execute their assigned tasks in parallel.

### 4. Module 1, Lesson 4: Introduction to GPU Architecture

#### 4.1 Hierarchical Components and Terminology

To understand the GPU, one must distinguish between the high-level organizational units and the low-level execution hardware.

- **Streaming Multiprocessor (SM):** In NVIDIA terminology, the SM is the equivalent of a "core" in traditional CPU nomenclature. It is a complex unit containing its own **Instruction Cache**, **Decoder**, **Shared Memory**, and multiple execution units.
- **Stream Processor (SP):** These are the individual **ALU (Arithmetic Logic Unit) units** or **SIMT lanes** within an SM. While often marketed as "cores" by manufacturers, for the purposes of high-level architectural study, they are categorized as lanes of a wider execution engine.
- **Core Definition:** In this curriculum, the term "core" refers to the Streaming Multiprocessor (SM) rather than the individual Stream Processor (SP) to avoid confusion between independent instruction streams and parallel lanes.

#### 4.2 The GPU Microarchitectural Pipeline

The GPU pipeline is designed to manage massive concurrency through a multi-stage process focused on throughput:

1. **Fetch Stage:**
    - Instructions are retrieved for specific work items.
    - The architecture utilizes **large register files** to support thousands of active threads simultaneously without the overhead of traditional context switching.
2. **Scheduling Stage:**
    - Sophisticated hardware schedulers determine the order of execution.
    - **Round-Robin Scheduler:** Rotates through ready tasks to ensure fairness.
    - **Greedy Scheduler:** Prioritizes tasks based on performance-critical factors such as **cache misses** or **branch prediction** outcomes to minimize stall time.
3. **Decode and Register Read:** Instructions are interpreted, and the necessary values are retrieved from the register file.
4. **Warp Selection (Scoreboarding):** A specialized scheduling mechanism (Scoreboarding) identifies **ready warps**—groups of threads whose operands are fully available—for immediate execution.
5. **Execution and Write-back:** The selected warps are processed by the execution units (ALUs), and the resulting data is committed back to the registers or memory.

#### 4.3 The Fundamental Unit of Execution: Warp/Wave-front

The **Warp** (NVIDIA term) or **Wave-front** (AMD term) represents the smallest unit of scheduled work in the GPU.

- **Structure:** A warp typically consists of **32 individual threads** that execute a single instruction in unison (SIMT).
- **Warp Size Stability:** The warp size has remained at 32 for several generations, though this is a microarchitectural parameter that could change in future designs.
- **Execution Logic:** The programmer specifies the total number of threads for a kernel (e.g., 12 threads). The hardware then groups these into warps based on its internal size (e.g., if warp size were 4, 12 threads would form 3 warps). The hardware executes one entire warp at a time as soon as its source operands are ready.

### 5. Module 1, Lesson 5: History and Evolution of GPU Architectures

#### 5.1 The Legacy of the 3D Graphics Pipeline

Historically, GPUs were fixed-function accelerators dedicated to 3D rendering.

- **Original Structure:** Separate hardware units were dedicated to **Vertex Processing** (calculating geometry) and **Pixel Processing** (rendering 2D primitives into screen pixels).
- **Fixed Functionality:** These early units could only perform the specific mathematical operations required for graphics, offering no flexibility for general computing.

#### 5.2 The Shift to Programmability and Unified Architecture

The evolution toward modern GPUs involved a gradual transition from fixed units to programmable cores.

- **Programmable Shaders:** Shaders evolved from fixed-function units to programmable units, allowing developers more control over vertex and pixel manipulation.
- **Unified Shader Architecture:** Introduced with the **Tesla architecture**, this innovation merged vertex and pixel processors into a single type of programmable core capable of handling any task.
- **Emergence of GPGPU:** The ability to use these programmable cores for non-graphics tasks gave birth to **General-Purpose Graphics Processing Unit (GPGPU)** programming, essentially treating the GPU as a massive parallel math co-processor.

#### 5.3 Modern Architectural Innovations

Recent GPU generations have expanded their capabilities to support High-Performance Computing (HPC) and Artificial Intelligence (AI):

- **Memory Advancements:** Integration of **High Bandwidth Memory (HBM)** directly into the GPU subsystem to provide unprecedented data transfer rates.
- **Cache Hierarchy:** Introduction of robust **L1 and L2 cache** structures to reduce memory latency.
- **Numerical Precision Scaling:**
    - Expansion from FP32 to **FP64 (Double Precision)** for scientific HPC applications.
    - Addition of smaller formats like **FP16** and **FP8** to accelerate Machine Learning (ML) workloads where lower precision is acceptable for speed.
- **Specialized Compute Cores:**
    - **Tensor Cores:** Specialized hardware for deep learning and matrix operations.
    - **Transformer Cores:** Specifically designed to accelerate Transformer-based ML models.
- **Expanded Instruction Sets:** Integration of atomic operations and fast integer operations to support diverse, non-graphical workloads.

#### 5.4 Case Study: NVIDIA H100 (2023)

The H100 represents the state-of-the-art in GPU architecture as of 2023, showcasing extreme scaling:

- **Multi-Precision Engine:** Supports FP64, FP32, FP16, FP8, and INT8 formats.
- **Transformer Engine:** A dedicated component for accelerating modern ML architectures.
- **Resource Scaling:** Features a significantly increased number of SMs and floating-point units compared to prior generations.
- **Memory Management:** Includes a **Tensor Memory Accelerator** and increased capacity for large register files.
- **System-Level Parallelism:** Utilizes the **NVLink switch system** to interconnect multiple GPUs, effectively creating a single, massive parallel processing entity.
