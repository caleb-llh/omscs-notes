### Phase 1: CPU vs. GPU Paradigm Split

At a fundamental level, a computer architecture is shaped entirely by what it optimizes for: **Latency** or **Throughput**.

#### 1. Latency-Sensitive (The CPU Approach)

A CPU is designed like a high-speed courier. Its primary directive is to execute a single, linear thread of instructions as quickly as humanly possible.

- **Precise Exceptions:** CPUs must support precise exceptions. If an instruction causes a fault (like a divide-by-zero or a page fault), the CPU must guarantee that all preceding instructions have fully completed, and no subsequent instructions have altered the architectural state. This requires enormous, power-hungry structural tracking like Reorder Buffers (ROB).
    
- **Role:** It acts as the **Host**—managing the Operating System, handling unpredictable I/O, and making complex structural branch decisions.
    

#### 2. Throughput-Sensitive (The GPU Approach)

A GPU is designed like a massive cargo train. It does not care if an individual piece of data takes a long time to cross the chip, as long as millions of pieces of data finish at the same time.

- **Imprecise Exceptions:** Because tracking exactly which thread caused a failure out of thousands of active threads would collapse performance, GPUs traditionally do not prioritize precise exceptions.
    
- **Role:** It acts as an **Accelerator**. It cannot independently run an operating system efficiently; it requires the CPU host to offload large, structured data blocks to it.
    

#### 3. The Instruction Set Architecture (ISA) Dichotomy

- **CPUs use Open/Public ISAs:** Architectures like x86 or ARM compile directly to a standardized machine code that runs natively on the hardware.
    
- **GPUs use Closed/Virtual ISAs:** GPU hardware changes so rapidly between generations that compiling directly to raw machine code would break software compatibility every year. Instead, programs compile to a _Virtual ISA_ (such as NVIDIA’s **PTX**—Parallel Thread Execution). At runtime, the GPU driver translates this virtual code into the actual, secret machine instructions specific to that exact physical silicon.
    

### Phase 2: Processing Paradigms (SISD, SIMD, SPMD, MIMD)

To classify these machines, the text maps them directly to **Flynn’s Classical Taxonomy**, which categorizes hardware by how it handles instruction streams versus data streams.

#### 1. SISD (Single Instruction, Single Data)

The classic, traditional single-core processor. One instruction manipulates one piece of data per cycle.

#### 2. SIMD (Single Instruction, Multiple Data)

A single execution unit controls multiple data paths simultaneously.

- **The Hardware Reality:** Think of this as vector extensions on a CPU (like AVX-512). There is exactly **one** master instruction decoder. It broadcasts a single instruction (e.g., `VECTOR_ADD`) to a wide arithmetic logic unit (ALU) that processes a fixed vector array (like 8 floating-point numbers) in absolute lockstep.
    
- **The Limitation:** It is highly rigid. If your data has an `if/else` branch, the hardware struggles because all lanes _must_ do the exact same thing at the exact same time.
    

#### 3. SPMD (Single Program, Multiple Data)

This is a **software programming model** favored by GPUs (and implemented via languages like CUDA or OpenCL).

- **The Paradigm Shift:** Instead of writing explicit vector instructions, the programmer writes code as if it is running on a _single, standard scalar thread_ (a "Kernel"). The runtime environment then instantiates thousands of copies of this exact same program, distributing them across massive datasets.
    
- **The Distinction:** Each thread has its own unique Thread ID (`index`), allowing it to dynamically calculate which piece of data it is responsible for. This provides the flexibility of writing normal code while achieving massive horizontal parallelism.
    

### Phase 3: The Deep Anatomy of a CPU Core

To understand why a GPU is structured the way it is, we must first look at the complexity a CPU undergoes to accelerate a single thread. A high-performance CPU core features a deeply pipelined, superscalar, out-of-order execution engine.

#### 1. The Five Fundamental Structural Stages

1. **Front-End / Fetch:** Instructions are pulled out of the Instruction Cache (I-Cache). High-performance CPUs use complex branch predictors here to guess which way an `if/else` statement will go before it even executes.
    
2. **Decode:** The raw bytes are translated into executable instructions. On complex architectures like x86, a single hardware instruction is decoded and broken down into multiple smaller, simpler operations called **micro-ops (uops)**.
    
3. **Rename & Schedule:** To bypass artificial data hazards (like writing to the same register name twice), the CPU maps software registers to a much larger pool of physical registers. The **Scheduler** then evaluates the instructions.
    
    - _In-Order:_ Executes strictly in the order of the source code.
        
    - _Out-of-Order (OoO):_ Searches the instruction window for _any_ instructions whose data dependencies are met, running them out of sequence to keep the execution units fully saturated.
        
4. **Execution:** The actual math occurs in the ALUs, or the memory sub-system is queried via Load/Store units.
    
5. **Write-Back / Commit:** The results are safely retired back into the architectural register file in the original program order, ensuring precise exceptions are preserved.
    

#### 2. Scaling CPU Parallelism

To push performance beyond one instruction per cycle (IPC > 1), CPUs rely on two structural paradigms:

- **Superscalar Execution:** Duplicating the front-end and execution pipelines so the core can fetch, decode, and execute _multiple independent instructions_ simultaneously per clock cycle. This relies purely on exploiting **Instruction-Level Parallelism (ILP)**.
    
- **Hardware Multi-Threading:** If a thread stalls waiting for data from slow main memory, the CPU core can instantly switch its execution context to a completely different thread. This requires duplicating the basic architectural state (like program counters and registers) but does not require duplicating the expensive execution ALUs.

### Phase 4: Mathematical and Logical Scaling limits (Amdahl’s Law)

When architects shift from the latency-optimized CPU model to the throughput-driven GPU model, the limits of parallelism are bounded by a brutal mathematical truth: **Amdahl's Law**.

Amdahl’s law governs the theoretical speedup of a program when utilizing a parallel computing infrastructure. It is expressed via the formula:

$$\text{Speedup} = \frac{1}{\left(\frac{P}{N} + S\right)}$$

Where:

- $P$ represents the fraction of the program that can be completely parallelized.
    
- $S$ represents the strictly serial portion of the program ($1 - P$).
    
- $N$ is the total number of parallel processing elements (cores or threads).
    

#### The "Serial Fraction" Bottleneck

If an application has an abundant parallel portion ($P$ is near 1.0) and the serial portion ($S$) is virtually zero, performance scales almost perfectly linearly as you add processors.

However, if even **10%** of your application is strictly serial ($S = 0.10$), the maximum theoretical speedup is mathematically hard-capped. Even if you throw an _infinite_ number of GPU cores at the problem ($N \to \infty$), the term $\frac{P}{N}$ drops to zero, leaving:

$$\text{Max Speedup} = \frac{1}{0.10} = 10\times$$

This underpins the entire philosophy of GPU programming: to extract any value out of thousands of parallel cores, algorithms must be aggressively restructured to minimize serial fractions and maximize structural data decomposition.

#### Task vs. Data Decomposition

To convert a standard software workload into parallel execution tracks, programmers rely on two core strategies:

1. **Task Decomposition:** Different execution units perform completely different operations on the same or distinct data. For example, in a game engine, Core 0 handles physics calculations while Core 1 simultaneously calculates an array's min/max value. This strategy scales poorly because a program naturally runs out of distinct tasks.
    
2. **Data Decomposition:** This is the bedrock of GPU computing. The exact same operation is applied uniformly across massive, fractured blocks of data. For instance, rather than having one thread sequentially loop through and add two arrays of 1 million elements, data decomposition splits the array into chunks, assigning an individual element-addition task to a million independent threads operating simultaneously.
    

### Phase 5: Inside the GPU—The Streaming Multiprocessor (SM)

To execute data-decomposed programs efficiently, the hardware layout must eliminate the massive overhead found in a CPU. In a GPU, we abandon complex out-of-order schedulers, huge branch predictors, and giant single-thread execution units. Instead, thousands of tiny arithmetic logic units are packed into clusters.

#### Demystifying GPU Core Terminology

GPU manufacturers use highly branded terms that can cause immense architectural confusion. The course clears this up by establishing strict definitions:

- **The Streaming Multiprocessor (SM) / SIMT Multiprocessor:** In the world of computer architecture, the SM is the actual "core" equivalent to a CPU core. It is an independent processing unit containing its own instruction cache, branch units, warp schedulers, and shared memory.
    
- **The Stream Processor (SP) / CUDA Core / SIMT Lane:** These are **not** independent cores. An SP is merely an individual Arithmetic Logic Unit (ALU) capable of performing basic math operations (like floating-point or integer calculations). An individual SM contains dozens or hundreds of these execution lanes.
    

#### The Layout of an SM

Within a single Streaming Multiprocessor, you will find:

- An **Instruction Cache** and hardware instruction decoders.
    
- A massive **Register File** designed to store the processing states of thousands of concurrent threads without needing to swap them to main memory.
    
- **Warp Schedulers:** Hardware engines that analyze collections of active threads and dispatch instructions to ready execution lanes.
    
- **Shared Memory / L1 Cache:** A high-speed, scratchpad memory shared locally by the execution units within that specific SM.
    

### Phase 6: The Unit of Execution—The Warp / Wavefront

GPUs do not manage or schedule threads individually; doing so would create a massive control-logic bottleneck. Instead, the hardware groups individual threads into fixed-size execution blocks.

- **Warp (NVIDIA terminology) / Wavefront (AMD terminology):** This is the fundamental, indivisible unit of hardware scheduling. In almost all modern GPU architectures, a warp consists of exactly **32 tightly coupled threads** (or 64 in some AMD architectures).
    

#### How a Warp Executes (The SIMT Model)

When a programmer writes a kernel, they might launch a grid containing 1,024 threads. The hardware instantly takes those 1,024 threads and fractures them into 32 distinct warps (Each warp containing 32 threads: Threads 0–31 form Warp 0, Threads 32–63 form Warp 1, etc.).

1. **Single Instruction Fetch:** For any given clock cycle, the SM fetches and decodes **exactly one instruction** for the entire warp.
    
2. **Lockstep Broadcast:** That single instruction is broadcast simultaneously across 32 individual SIMT lanes (ALUs).
    
3. **Differentiated Execution:** Every thread in that warp executes that exact same instruction at the exact same instant, but each thread uses its own unique registers and its own internal Thread ID to manipulate a different piece of data.
    

#### The GPU Hardware Pipeline

The execution pipeline of a GPU core runs a continuous cycle:

1. **Fetch & Scoreboarding:** The front-end fetches instructions for the active warps. A hardware mechanism called a **Scoreboard** tracks which warps are stalled (waiting for a memory read to return from RAM) and which warps have all their data operands ready in the register file.
    
2. **Scheduler Selection:** The Warp Scheduler applies a selection algorithm—such as simple _Round-Robin_ or a latency-hiding _Greedy Scheduler_—to select a warp that is fully ready to execute.
    
3. **Decode & Dispatch:** The instruction for the chosen warp is decoded, register operands are pulled out of the massive register file, and the instruction is dispatched directly to the array of stream processors/SIMT lanes.
    
4. **Write-Back:** The math results are committed directly back to the register file.
    

By grouping threads this way, a GPU can use a single, tiny instruction decoder to control 32 ALUs at once. This frees up massive amounts of silicon space, allowing the manufacturer to remove CPU control logic and replace it with more raw, mathematical computing muscle.

### Phase 7: The Evolutionary History of GPU Shaders

The modern general-purpose GPU (GPGPU) did not emerge fully formed; it evolved from a highly rigid, fixed-function hardware layout designed entirely for rendering 3D pixels.

```
[Traditional GPU] ──> Fixed-Function Units (Separate Vertex & Pixel Processors)
                             │
                             ▼
[Programmable GPU] ─> Unified Shader Architecture (Tesla: Cores do both jobs)
                             │
                             ▼
[Modern GPGPU] ────> Hardware Accelerators (Tensor Cores, Ray Tracing, DPUs)
```

#### 1. Traditional 3D Graphics Pipeline

Early GPUs used separate, dedicated hardware pipelines for different stages of the graphics rendering process:

- **Geometry/Vertex Processing:** Mathematical units dedicated purely to calculating 3D coordinates (vertices) of primitives.
    
- **Rasterization & Pixel Processing:** Completely separate hardware units designed to translate those 3D coordinates into 2D pixels on a screen and color them.
    
- **The Bottleneck:** This separate architecture was highly inefficient. If a game required complex geometric calculations but simple coloring, the vertex processors would be completely overloaded while the pixel processors sat idle.
    

#### 2. The Shift to Unified Shaders (The GPGPU Era)

To solve this utilization imbalance, architectures transitioned to a **Unified Shader Architecture** (pioneered by architectures like NVIDIA’s Tesla).

- Instead of separate hardware for vertices and pixels, the GPU was redesigned as a massive collection of identical, highly programmable computing pipelines.
    
- A single stream processor could calculate a 3D vertex position during one cycle, and calculate a pixel's color value during the next cycle.
    
- This uniform layout made it possible to use the GPU for tasks completely unrelated to graphics. Software developers realized they could map raw mathematical matrices to these unified shaders, sparking the modern era of General-Purpose GPU computing.
    

### Phase 8: Modern GPU Memory and Architectural Enhancements

As processing speeds outpaced memory performance, GPUs underwent radical architectural upgrades to keep their massive ALU arrays fed with data.

#### 1. Advanced Cache Hierarchies

Early GPUs lacked traditional CPU-style caches because they relied purely on huge numbers of threads to hide memory latency. Modern GPUs integrated sophisticated L1 and L2 cache hierarchies directly onto the die to reduce the penalty of frequently accessing off-chip VRAM.

#### 2. High Bandwidth Memory (HBM)

Standard GDDR memory chips sit around the edges of a GPU on a traditional circuit board, communicating across narrow, power-hungry buses.

- **HBM Design:** High Bandwidth Memory stacks DRAM dies vertically on top of each other, connecting them with thousands of microscopic vertical wires called Through-Silicon Vias (TSVs).
    
- **The Silicon Interposer:** This stack sits on a ultra-fine silicon layer right next to the main GPU die, inside the same physical chip package. This allows for an extraordinarily wide memory bus (thousands of bits wide compared to GDDR's 256-bit or 384-bit buses), unlocking unprecedented data transfer rates.
    

#### 3. Mixed Precision and Machine Learning Formats

Traditional high-performance computing (HPC) requires high-precision 64-bit floating-point math (FP64) to ensure absolute scientific accuracy. However, training deep neural networks does not require that level of precision.

- To maximize computational throughput, modern GPUs added native support for lower-precision formats like **FP16** and **INT8**.
    
- Because an FP16 number takes up half the space of an FP32 number, the hardware can perform twice as many mathematical operations per clock cycle using the exact same silicon area, drastically accelerating machine learning training loops.

#### 1. Tensor Cores

Standard streaming multiprocessors process vectors (one-dimensional arrays of numbers). **Tensor Cores** are highly specialized hardware accelerators built inside the SM designed to calculate entire _matrices_ (two-dimensional blocks of numbers) in a single instruction. They calculate:

$$D = A \times B + C$$

This exact matrix multiply-accumulate operation is the core mathematical building block of all deep learning layers.

#### 2. The Transformer Engine

The Hopper architecture introduced a hardware-software layer known as the Transformer Engine. It dynamically analyzes the layer of an AI model during training and automatically switches the precision format down to highly efficient **FP8 (8-bit floating point)** when the model can tolerate it, dropping back to FP16 only when numerical stability is threatened. This provides a massive boost in training volume without sacrificing final accuracy.

#### 3. Scaling out via NVLink Switch Systems

A single H100 chip contains an immense number of streaming multiprocessors, but a single chip is still not large enough to train modern large language models.

- To scale further, the hardware integrates dedicated **NVLink** controllers.
    
- Instead of routing traffic through the slow, standard PCIe lanes of the host server, NVLink creates an ultra-high-speed, proprietary direct mesh network between multiple GPUs.
    
- This interconnect treats a cluster of separate physical GPUs across a server rack as one unified, massive computing fabric, eliminating the traditional network bottlenecks of distributed clusters.
    

### Core Course Summary

This entire architectural journey bridges the foundational concepts we explored earlier to the design of modern cloud data centers:

1. **The System Layer:** We start with **SMP** (Symmetric Multiprocessing) where multiple cores act as equal peers, using **NUMA** (Non-Uniform Memory Access) to break up the memory highway and avoid massive system bus bottlenecks.
    
2. **The Chip Layer:** Inside those processors, we manage traffic using **Network-on-Chip (NoC)** mesh topologies, relying on **Distributed LLC Slicing** and **Partial Directories** to track data states without wasting precious silicon space.
    
3. **The Primitive Layer:** Cores use atomic hardware primitives like **LL/SC** (Load-Linked/Store-Conditional), leveraging the built-in cache coherency protocol to provide non-blocking synchronization.
    
4. **The Accelerator Layer:** When applications demand pure volume, we shift from latency-optimized CPU cores to throughput-optimized **Many-Core GPUs**, trading complex control logic for thousands of simple execution lanes (SIMT) grouped into warps.
    
5. **The Network Layer:** Finally, at the data center scale, **DPUs** offload the operating system's communication tax, allowing memory networks to act as a fully disaggregated, scalable computing fabric.
