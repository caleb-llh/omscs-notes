## II. GPU Cycle-Level Simulation (Part 1)

### 2.1 Overview of Performance Modeling Techniques

There are several established methodologies for modeling hardware performance:

- **Cycle-Level Simulation:** Models hardware state transitions on a cycle-by-cycle basis.
- **Event-Driven Simulation:** Progresses based on discrete events rather than a global clock.
- **Analytical Modeling:** Uses mathematical formulas to predict performance.
- **Sampling-Based Techniques:** Simulates representative portions of a workload to extrapolate total performance.
- **Statistical and Machine Learning (ML) Modeling:** Uses database-driven or ML-based approaches to predict architectural behavior.
- **FPGA-Based Emulation:** Implements logic on reconfigurable hardware for high-speed verification.

### 2.2 Cycle-Level Simulation Mechanics

In cycle-level simulation, a **global clock** synchronizes all components.

- **Clock Domains:** Simulators can model multiple asynchronous domains, including memory, processor, and Network-on-Chip (NoC) clocks.
- **Simulation Types:**
    - **Execution-Driven:** Instructions are executed during simulation, allowing for the modeling of runtime-dependent behaviors like lock acquisition and barriers.
        - _Execute-at-Fetch:_ Instructions execute as soon as they are fetched.
        - _Execute-at-Execute:_ Instructions execute at the functional unit stage.
    - **Trace-Driven:** Execution is decoupled from simulation. Traces are collected beforehand.
        - _Benefits:_ Simpler, lighter, and easier to develop; useful for isolated memory or cache studies.
        - _Limitations:_ Cannot model dynamic runtime behaviors or synchronization primitives.

### 2.3 Queue-Based Pipeline Modeling

Queue-based modeling is the standard for representing a 5-stage pipeline.

- **Structural Components:**
    - **FIFOs:** Used between most pipeline stages (Fetch, Decode, Execution, Write-Back) to model buffering.
    - **Scheduler:** Unlike other stages, the scheduler is not a FIFO; it selects instructions from a pool of "ready" candidates based on resource availability.
- **Modeling Parameters:**
    - **Pipeline Latency:** Represented by the **depth** of the queue.
    - **Pipeline Bandwidth/Width:** Represented by the number of instructions allowed to move between queues per cycle.
    - **Instruction Latency:** Defined as fixed parameters (e.g., ADD = 1 cycle, MUL = 3 cycles) derived from literature, CACTI, or RTL documentation.

### 2.4 Case Study: Macsim Simulator Structure

Macsim is a hierarchical CPU/GPU simulator developed at Georgia Tech.

**A. Simulation Loop Logic:**

1. Instantiate simulator modules.
2. Initialize parameters.
3. Execute `run_a_cycle()` iteratively until the program termination condition is met.
4. Hierarchy: `macsim::run_a_cycle` $\rightarrow$ `CPU/GPU/Memory/NoC::run_a_cycle` $\rightarrow$ `Pipeline::run_a_cycle`.

**B. The `op` Data Structure:** Each instruction is tracked via an `op` structure that records its progress through the pipeline.

- **Done_cycle Calculation:** `done_cycle = schedule_time + instruction_latency`.
- **Variable Latency:** For events like cache misses, the simulator queries the cache model to determine the appropriate latency before updating the `done_cycle`.

### 2.5 GPU-Specific Cycle-Level Modeling

- **Execution Unit:** The simulation unit is a **warp** (not a thread) to reflect SIMT behavior.
- **Scheduling:** Warps are scheduled in-order internally, but the SM can execute warps out-of-order relative to each other.
- **Simulation Termination Hierarchy:**
    1. Individual threads complete.
    2. All threads in a CUDA block complete $\rightarrow$ **Block Retirement**.
    3. All blocks in a kernel complete $\rightarrow$ **Kernel Completion**.
    4. All kernels in an application complete $\rightarrow$ **Application End**.

## III. GPU Cycle-Level Simulation (Part 2)

### 3.1 Advanced Modeling Units

- **Warp-Level Modeling:** Instructions are tracked at the warp level, including Instruction Cache (I-cache) and Program Counter (PC) register accesses.
- **Mask Bits:** Used to track which threads within a warp are active for a given instruction to enforce resource constraints.

### 3.2 Branch Divergence Modeling

- **SIMT Stack:** Execution-driven simulators must faithfully model the hardware SIMT stack to handle different execution paths.
- **Trace-Driven Divergence:** Traces must contain mask bit information for every Basic Block (BB).
    - _Example:_ BB#1 (mask: 1111) $\rightarrow$ BB#2 (mask: 1100) and BB#3 (mask: 0011) $\rightarrow$ BB#4 (Re-convergence mask: 1111).

### 3.3 Memory Coalescing and Cache Modeling

- **Coalescing Logic:** Crucial for determining the actual number of memory transactions and bandwidth consumed.
- **Transaction Granularity:** Typically follows the cache line size (e.g., 64 bytes).
    - _Coalesced:_ Multiple thread requests merged into one 64-byte transaction.
    - _Uncoalesced:_ Each thread request generates an independent transaction (e.g., thirty-two 64-byte requests for one warp instruction).
- **Sectored Caches:** Modern GPUs use sectored caches to bring in only a portion of a block (e.g., a 16-byte sector instead of the full 64-byte line) to conserve bandwidth, though this may reduce spatial locality.

## Part 4: Analytical Models of GPUs (Module 8, Lesson 3)

### 4.1 Fundamentals of Analytical Modeling

Analytical models provide a **first-order estimation** of processor design parameters and application performance without requiring the execution of an entire program. They are prized for being simple, insight-driven, and capable of rapid design space exploration.

### 4.2 First-Order GPU Architecture Design Exploration

Analytical models are used to determine hardware requirements based on specific performance goals.

- **Design Goal Case Study: 1T Vector Dot Product/sec**
    
    - **Equation:** `sum += x[i] * y[i]`.
    - **Compute Requirements:** To achieve 1T dot products, the system requires **2T FLOPS** (separate multiply and add) or **1T FMA/sec** (Floating Point Multiply-Add).
    - **Hardware Scaling:**
        - At 1 GHz: Requires 1,000 FMA units.
        - At 2 GHz: Requires 500 FMA units.
    - **Memory Bandwidth Requirements:** Must supply two memory bytes (for `x` and `y`) per FMA, requiring **2 TB/sec** of bandwidth.
- **Multithreading Design Parameters ($W_{width}$ and $W_{depth}$):**
    
    - **$W_{width}$:** The number of threads executing in a single cycle (e.g., 32). Determined by ALU units and scheduler width.
    - **$W_{depth}$:** The maximum number of unique threads/warps that can be scheduled during one stall cycle. Determined by the number of registers and PC registers.
    - **Hardware Constraints:** Total registers needed = $W_{depth} \times W_{width} \times \text{registers per thread}$.
- **SM and Resource Decisions:**
    
    - **Configuration Trade-offs:** Total execution bandwidth can be achieved via few large SMs or many small SMs (e.g., 16 SMs with 1 warp/cycle vs. 8 SMs with 2 warps/cycle).
    - **Cache/Register Impact:** Many small SMs result in small caches per SM, which can increase access time or affect hit rates if data is shared across thread blocks.
    - **Sub-core Architecture:** A design where multiple sub-cores (each with its own PC, decoder, and warp scheduler) share common I-cache and large L1 caches within a single SM.

### 4.3 The Roofline Model

The Roofline model is a visual tool to determine if an application or processor is limited by **compute bandwidth** or **memory bandwidth**.

- **Axes:** Y-axis represents performance; X-axis represents **Arithmetic Intensity** (FLOPS per byte).
- **Performance Bounds:**
    - **Memory Bounded:** The sloped portion of the "roof," where performance is limited by memory bandwidth.
    - **Compute Bounded:** The flat peak of the "roof," representing peak hardware FLOPS.
- **Example (Arithmetic Intensity):**
    - Vector Sum: 2 bytes per 1 FLOP = 0.5 intensity (Memory Bounded).
    - Sum with Squares ($sum += x[i]*x[i] * y[i]*y[i]$): 2 bytes per 4 FLOPS = 2.0 intensity (moves toward Compute Bounded).

### 4.4 CPI (Cycles Per Instruction) Computation

- **Standard Formula:** $Average\ CPI = CPI_{steady_state} + \sum CPI_{event_n}$.
- **Serialized Penalty Example (5-stage in-order):**
    - $CPI_{steady_state} = 1$
    - Branch misprediction (2% freq, 3 cycle penalty)
    - Cache miss (5% freq, 5 cycle penalty)
    - $Average\ CPI = 1 + (0.02 \times 3) + (0.05 \times 5) = 1.31$.
- **Multi-threading Extension:** Ideally, $CPI_{multi-threading} = \frac{CPI_{single_thread}}{W_{depth}} + \text{Resource Contention}$. Contentions include MSHR limits, busy execution units, and DRAM bandwidth.

### 4.5 Interval-Based Modeling

This is an extension of CPI modeling that defines intervals containing one steady state and one long-latency event.

- **Mechanism:** Models the "drain" and "refill" time of a pipeline rather than assuming an abrupt stall.
- **GPU Considerations:** To accurately model GPUs via interval analysis, two divergence factors must be added:
    1. **Branch Divergence:** Accounts for serialized branch paths.
    2. **Memory Divergence:** Accounts for latency differences between coalesced and uncoalesced requests.

## Part 5: Accelerating GPU Simulation (Module 8, Lesson 4)

### 5.1 Challenges of Cycle-Level Simulation

Cycle-level simulation is extremely time-intensive. Simulating at 1 kilo-instructions per second (KIPS) requires approximately **28 hours** to model just **one billion instructions** (roughly one second of real-time execution). Complex workloads like machine learning could take "centuries" of simulation time if not accelerated.

### 5.2 Simulator Acceleration Techniques

- **Parallelizing the Simulator:** Assigning each thread to model a specific core.
    - _Challenge:_ Communication and memory modeling often require serialization, becoming performance bottlenecks, especially in memory-bounded workloads.
- **Event-Driven Simulation:** Operates based on a queue of discrete events rather than checking every cycle. This is highly efficient for modeling long-latency operations like I/O or NVM (Non-Volatile Memory) accesses.
- **Model Simplification:** Reducing accuracy for speed.
    - _Examples:_ Assuming IPC equals issue width and only modeling memory; using fixed memory latencies when evaluating compute logic.

### 5.3 Sampling Techniques

Sampling involves simulating only representative portions of a workload to extrapolate total performance.

- **General Methods:** Random sampling, execution-driven (skipping non-essential parts), and trace-driven sessions.
- **State Warming:** Critical for cache and branch predictors; the simulator must "warm up" these structures to ensure results are not skewed by initialization effects.
- **Program Phase Detection (Simpoint):** Identifies distinct phases (e.g., High IPC vs. Low IPC) and simulates only one instance of each representative phase.
- **GPU-Specific Sampling:**
    - **Block-Level:** Simulating a subset of CUDA blocks (e.g., 100 instead of 1,000).
    - **Kernel-Level:** Simulating representative kernels, useful for ML workloads that repeat the same kernel hundreds of times.
    - **Warp-Level:** Reducing the number of warps simulated per SM.

### 5.4 Workload Reduction

- **Iteration Reduction:** Simulating a single training iteration instead of thousands in ML workloads.
- **Input Scaling:** Reducing graph nodes or data array sizes (e.g., from 1 billion nodes to 1 million).
- **Kernel Focus:** Identifying and simulating only the one or two kernels that dominate 90% of execution time.

### 5.5 Data-Driven and ML Modeling

Constructs models based on statistical data collected from hardware performance counters or simulators rather than underlying architectural logic.

- **Feature Extraction:** Identifying variables that explain performance (IPC, instruction count, number of loads, divergent branches).
- **Predictive Formula Example:** $Total\ Cycle\ Count = \sum (C_n \times EventCount_n^{exponent})$.
    - _Example:_ $Cycles = (C_1 \times InstrCount^{E1}) + (C_2 \times FP_InstrCount^{E2}) + (C_3 \times Mem_InstrCount^{E3})$.
- **Implementation:** Coefficients ($C$) and exponents ($E$) are derived using machine learning or statistical analysis on collected training data.
