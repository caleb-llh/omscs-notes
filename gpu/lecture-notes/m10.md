# **GPU Architecture and Compilation Manual: Part 1**

This manual provides a comprehensive structural synthesis of the GPU compiler flow and Parallel Thread Execution (PTX) architecture based on the technical documentation provided.

## **I. GPU Compiler Flow**

### **1.1 Fundamentals of Compilation**

The standard compilation process consists of three primary stages:

- **Front-end Compiler:** Converts source code into an **Intermediate Representation (IR)**.
- **Compiler Optimizer:** Executes various optimizations directly on the IR.
- **Backend Code Generation:** Performs machine-specific target optimizations and emits the final target program.

### **1.2 GPU-Specific Compilation Characteristics**

The GPU compilation flow follows the standard model but includes unique steps to handle the heterogeneous nature of the code:

- **Code Splitting:** The source code is divided into **host code** (running on the CPU) and **device code** (running on the GPU).
- **NVIDIA-Specific Path:** For NVIDIA GPUs, the device code is converted into **NV PTX code**.
- **Fat Binaries:** The PTX assembly is combined with both host and device code to create a **fat binary**. This binary contains execution files for multiple GPU architectures and is split again at runtime by the device driver.
- **Linking:** Linking is a crucial step due to NVIDIA's use of runtime libraries and various supporting libraries that run on the host.
- **Virtual Architecture:** Because PTX is a virtual Instruction Set Architecture (ISA), the underlying architecture is referred to as a virtual architecture.

### **1.3 Toolchain Components**

- **CLANG:**
    - Serves as the front-end parser and a toolchain for C-family languages (now extended to others).
    - Generates the **Abstract Syntax Tree (AST)**, which represents the hierarchical structure of the code.
    - Example: A simple code block computing "delta" can generate a highly complex AST node structure depending on its usage.
- **C++ PreProcessor:**
    - Performs text substitution prior to compilation.
    - **Example:** `#define course Number 8803` results in the string "course Number" being replaced by "8803" throughout the source.
- **IR (Intermediate Representation) Optimizations:**
    - Occur within the backend compiler process.
    - IR allows compiler infrastructure to be shared across different programming languages and hardware architectures.
    - Contains metadata to carry program information that assists the compiler.

### **1.4 Machine Code: PTX vs. SASS**

- **PTX (Parallel Thread Execution):**
    - A **virtual ISA** that is architecture-independent.
    - Does **not** have register allocation because it lacks specific target machine information.
    - Remains consistent across different GPU generations.
- **SASS (Shader Assembly):**
    - A **real low-level assembly language** that is architecture-dependent.
    - Includes target information, meaning **registers are explicitly allocated**.
    - SASS code changes when moving to a new or different GPU architecture.

## **II. PTX (Parallel Thread Execution) Architecture**

### **2.1 Instruction Format**

PTX instructions typically contain between **zero and four operands**. The format supports optional **predicate information** indicated by the `@` symbol.

- **General Syntax:** `[@p] opcode d, a, b, c;` where `d` is the destination and `a, b, c` are sources.
- **Special Instruction (`setp`):** This instruction writes to two destination registers, separated by a vertical bar (`|`).
    - **Example:** `setp.lt.s32 p | q, a, b;`
    - **Logic:** Sets `p` to true if `a < b`, and sets `q` to the opposite (NOT) of `p`.

### **2.2 Predicated Execution**

GPUs utilize predication to eliminate **divergent branches**.

- **Predicate Registers:** Must be declared using the `.reg .pred` specifier (e.g., `.reg .pred p, q, r`).
- **C++ to PTX Transformation Example:**
    - **C++ Code:** `if (i < n) { j = j + 1; }`.
    - **Equivalent PTX:**
        1. `setp.lt.s32 p, i, n;` (Sets predicate `p` if `i < n`).
        2. `@p add.s32 j, j, 1;` (Executes addition only if `p` is true).

### **2.3 PTX Statements and Directives**

A PTX program consists of **directives** (which set up the environment) or **instructions** (which perform work). Statements can start with a label and must end with a semicolon.

- **Common Directives:** `.target`, `.address_size`, `.func`, `.reg`, and `.global`.
- **Basic Instructions:** `mov` (move), `shl` (shift left), `ld` (load), and `add` (addition). All instructions must include a **type specifier** indicating the data type (e.g., `.s32`, `.u32`).

### **2.4 Specialized Instruction Categories**

- **Control Flow:**
    - `bra targ1`: Branch to label `targ1`.
    - `call func`: Call function named `func`.
    - `ret`: Return from a function.
- **Synchronization:**
    - `membar`: Memory barrier.
    - `fence`: Memory fence operation.
- **Atomic Operations:**
    - Identified by the `atom` prefix.
    - **Example:** `atom.add.f16` (Atomic addition for 16-bit floating point).
    - Supports both integers and floating-point types.
- **Special Registers:**
    - `%ntid`: Number of threads in a CTA (Cooperative Thread Array).
    - `%tid`: Thread ID.
    - `%sp`: Stack pointer.

# **GPU Architecture and Compilation Manual: Part 2**

This section details Intermediate Representation (IR) structures, the identification and construction of Basic Blocks, and the foundational principles of Data Flow Analysis.

## **III. Intermediate Representation (IR) and Code Structure**

### **3.1 IR Syntax and Standards**

Intermediate Representation typically utilizes **three-address code**, following the general format `A = B op C`.

- **LLVM IR Example:** `%result = add i32 %a, %b`.
    - `%result`: Destination register and target variable.
    - `add`: The operation performed.
    - `i32`: Type specifier indicating a 32-bit integer.
    - `%a, %b`: Source operands.
- **PTX IR Equivalent:** `add.u32 r1, r2, r3` or `add.s32 r1, r2, r3`.
    - PTX requires distinct opcodes for signed (`.s32`) and unsigned (`.u32`) integers.

### **3.2 Basic Block Definition**

A basic block is the **maximum sequence of instructions** characterized by exactly one entry point and one exit point.

- **Entry Rule:** Only the first instruction in the block can be reached from outside the block.
- **Execution Rule:** Once the program enters a basic block, all instructions within it must be executed consecutively.
- **Exit Rule:** The final instruction is typically a control flow instruction.
- **Optimization Scope:** Optimizations performed strictly within a basic block are termed **local code optimizations**.

### **3.3 Algorithm for Identifying Basic Blocks**

To partition code into basic blocks, the compiler follows a two-step process to identify "leaders".

**Step 1: Identify Leaders** An instruction is a leader if it meets any of the following criteria:

1. It is the **first instruction** in the program.
2. It is the **target** of a conditional or unconditional jump/branch.
3. It **immediately follows** a conditional or unconditional jump/branch.

**Step 2: Grouping** Instructions are grouped starting from one leader up to, but not including, the next leader. Each group constitutes a single basic block.

### **3.4 Control Flow Graphs (CFG)**

A CFG is a graphical representation where **nodes represent basic blocks** and **edges indicate possible execution paths**.

- **Entry Node:** The first statement or block of the program.
- **Path Logic:** If Block A can transition to Block B (e.g., via a branch or falling through), a path exists between them.
- **Predecessors and Successors:**
    - **Predecessor:** All blocks executed immediately before the current block.
    - **Successor:** All blocks executed immediately after the current block.

---

## **IV. Data Flow Analysis and Global Optimization**

### **4.1 Global Code Optimization**

While local optimization is confined to a single basic block, **global code optimization** occurs across multiple basic blocks.

- **Foundational Mechanism:** Most global optimizations are based on **Data Flow Analysis**, which tracks how program properties change across the CFG.
- **Preservation Requirement:** Compiler optimizations **must preserve the semantics** and functionality of the original program.

### **4.2 Common Optimization Techniques**

- **Redundant Instruction Removal:** Eliminating instructions that perform duplicate work.
- **Copy Propagation:** Identifying constant values and propagating them through the code, often revealing further redundancies.
- **Dead Code Elimination:** Removing code segments that will never be executed.
- **Code Motion:** Moving code to more efficient locations to improve performance.
- **Induction Variable Detection:** Identifying variables for specialized optimizations like **reduction strength**.

### **4.3 Mathematical Abstraction of Data Flow**

Data flow analysis treats program execution as a transformation of states.

- **Program Points:** The state "IN" is the point immediately before a statement; the state "OUT" is the point immediately after.
- **Transfer Functions ($f_B$):** These functions represent the effect of a basic block on the program state: $OUT[B] = f_B(IN[B])$.
- **Directional Logic:**
    - **Forward Flow:** $IN[B]$ is the union of the $OUT$ values of all predecessor blocks.
    - **Backward Flow:** $OUT[B]$ is derived from the $IN$ values of successor blocks.

# **GPU Architecture and Compilation Manual: Part 3**

This section details the mechanics of Reaching Definitions, the derivation of Gen and Kill sets, and the iterative algorithm used for global data flow analysis.

## **V. Reaching Definitions Analysis**

### **5.1 Definitions and Points**

Reaching definition analysis determines whether a specific definition of a variable reaches a certain point in the program without being altered.

- **Definition ($d$):** A point where a variable receives a value (e.g., `a = x + y` defines `a`).
- **Use:** A point where a variable's value is read (e.g., `a = x + y` uses `x` and `y`).
- **Reachability:** A definition $d$ reaches point $p$ if there is a path from the point immediately following $d$ to $p$ such that $d$ is not **"killed"** (overwritten) along that path.

### **5.2 Gen and Kill Logic**

Each statement $d$ (e.g., `u = v + w`) has an associated effect on the set of reaching definitions:

- **Gen Set ($gen_d$):** The set containing the definition $d$ itself.
- **Kill Set ($kill_d$):** The set of all other definitions in the program that define the same variable (e.g., $u$).
- **Transfer Function for Statement $d$:** $f_d(x) = gen_d \cup (x - kill_d)$.

---

## **VI. The Iterative Reaching Definitions Algorithm**

### **6.1 Equations and Constraints**

The compiler uses the following equations to solve for reaching definitions across the Control Flow Graph (CFG):

- **IN[B]:** The union of the `OUT` sets of all predecessors of block $B$.
    - $IN[B] = \bigcup_{P \in pred(B)} OUT[P]$
- **OUT[B]:** The result of applying the transfer function to the input.
    - $OUT[B] = gen_B \cup (IN[B] - kill_B)$
- **Boundary Condition:** $OUT[ENTRY] = \emptyset$ (empty set).

### **6.2 The Iterative Process**

1. **Initialization:** Set $OUT[ENTRY]$ to empty. For all other basic blocks $B$, initialize $OUT[B]$ to $\emptyset$ (or all zeros in a bit-vector representation).
2. **Iteration:**
    - For each basic block $B$, compute $IN[B]$ by combining $OUT$ values of predecessors.
    - Compute $OUT[B]$ using the transfer function.
3. **Convergence:** Repeat the iteration until the $OUT$ values for all blocks remain unchanged from the previous pass.

---

## **VII. Worked Illustration of Reaching Definitions**

The following example tracks seven definitions ($d1$ to $d7$) using a **7-bit vector** where each bit represents a definition.

### **7.1 Initial State (Iteration 0)**

- All $OUT[B]$ values are initialized to `0000000`.

### **7.2 Iteration 1 Walkthrough**

- **Basic Block 1 (B1):**
    - $IN[B1] = OUT[ENTRY] = 0000000$.
    - $gen_{B1} = {d1, d2, d3}$. $kill_{B1} = {d4, d5, d6, d7}$.
    - $OUT[B1] = 1110000$.
- **Basic Block 2 (B2):**
    - $IN[B2] = OUT[B1] \cup OUT[B4] = 1110000 \cup 0000000 = 1110000$.
    - $kill_{B2} = {d1, d2, d7}$. $gen_{B2} = {d4, d5}$.
    - $OUT[B2] = (1110000 - {d1, d2, d7}) \cup {d4, d5} = 0011100$.
- **Basic Block 3 (B3):**
    - $IN[B3] = OUT[B2] = 0011100$.
    - $kill_{B3} = {d3}$. $gen_{B3} = {d6}$.
    - $OUT[B3] = (0011100 - {d3}) \cup {d6} = 0001110$.
- **Basic Block 4 (B4):**
    - $IN[B4] = OUT[B2] \cup OUT[B3] = 0011100 \cup 0001110 = 0011110$.
    - $kill_{B4} = {d1, d4}$. $gen_{B4} = {d7}$.
    - $OUT[B4] = (0011110 - {d1, d4}) \cup {d7} = 0010111$.

### **7.3 Iteration 2 (Sample Logic)**

In the second iteration, $IN[B2]$ changes because $OUT[B4]$ is no longer zero.

- $IN[B2] = OUT[B1] (1110000) \cup OUT[B4] (0010111) = 1110111$.
- After applying the B2 transfer function (kill $d1, d2, d7$ and gen $d4, d5$), the new $OUT[B2]$ becomes `0011110`.
- The process continues until no further bit-flips occur.
