# GPU Technical Manual: Virtual Memory and Warp Scheduling

## Part 1: GPU Virtual Memory (Module 7, Lesson 1)

### 1.1 Fundamentals and Importance

Virtual memory is essential in modern GPU architectures to manage complex memory systems and improve programmability. It involves two primary aspects:

- **Address Translation:** Mapping virtual addresses assumed by programmers to physical addresses.
- **Page Allocations:** Managing physical memory chunks. While the CPU OS handles this for CPUs, GPUs initially rely on the CPU and GPU drivers using API functions like `cudaMalloc`, `cudaMallocManaged`, and `cudaMemcpy`.

### 1.2 Address Translation Components

- **VPN (Virtual Page Number):** The portion of the virtual address being translated.
- **PFN/PPN (Physical Frame/Page Number):** The resulting physical location mapping.
- **PO (Page Offset):** Remains constant during translation.
- **Standard Page Sizes:** Typically 4 KB or 2 MB.
- **PTE (Page Table Entry):** A memory entry storing the VPN-to-PFN mapping and metadata.
- **TLB (Translation Look-aside Buffer):** A cache for address translations (PTEs) that speeds up future accesses by avoiding multiple memory reads.
- **TLB Miss:** Occurs when a translation is not in the TLB, requiring a page table access.
- **Page Fault:** Occurs when no physical page is yet mapped, requiring the OS to allocate a page.

### 1.3 Multi-Level (4-Level) Address Translation

To avoid storing massive single-level tables (which would require $2^{48}$ entries for a 48-bit VPN), a hierarchical structure is used.

- **Structure:** Indices from Level 4 down to Level 1 are used to navigate the hierarchy.
- **The "Walk":** A single translation requires at least four memory accesses (a "walk").
    - Level 4 index accesses the Level 4 table to find the Level 3 starting address.
    - This repeats through Level 2 and Level 1.
    - The Level 1 entry contains the PTE with the PFN.
- **Implementation:** Handled by a hardware Page Table Walker (PTW) or via software.

### 1.4 GPU-Specific Virtual Memory Implementation

- **Evolution:** Early GPUs had no virtual memory or only one-to-one mapping; modern GPUs require complex support for features like Unified Virtual Address (UVA).
- **Pipeline Stage:** The memory address stage is split into **address coalescing** and **TLB accesses**.
- **Memory Coalescing Impact:**
    - **Coalesced Access:** One warp generates one memory request and one TLB access.
    - **Uncoalesced Access:** If requests cannot be combined, one memory instruction can generate multiple TLB accesses (potentially equal to the warp width).
- **Scalability Challenges:** A single warp TLB miss can trigger multiple subsequent misses, demanding high bandwidth from the PTW. Sharing a high-bandwidth PTW across multiple Streaming Multiprocessors (SMs) is one solution.

### 1.5 Optimization Techniques for Address Translation

- **Segment Mapping:** Assigning specific portions of physical address space to virtual address spaces (e.g., Program 1 gets 0x00, Program 2 gets 0x01). Translation involves simply adding upper bits, resulting in low cost.
- **TLB Hierarchies:** Implementing multiple levels of TLBs.
- **Large Pages (2 MB vs 4 KB):**
    - **Benefits:** Reduces VPN bits (e.g., from 51 to 42) and reduces the number of TLB entries needed by a factor of 512.
    - **Downside:** Increases memory fragmentation (e.g., needing only 1 KB but allocating 2 MB wastes nearly the entire page).

## Part 2: Unified Virtual Address Space (UVA) (Module 7, Lesson 2)

### 2.1 Global Memory Management Models

1. **Copy-then-Execute Model (Traditional):**
    - Requires explicit data transfer via `cudaMemcpy` after allocating with `cudaMalloc`.
    - Ensures data is available on the GPU before the kernel launches.
    - **Pros:** Efficient PCI-E bandwidth usage via bulk transfers.
    - **Cons:** Complex manual management for programmers.
2. **Unified Virtual Address Space (UVA) / Shared Virtual Memory (SVM):**
    - CPU and GPU share the same logical address space.
    - **Pros:** Significantly increases programmability; logically, only one buffer exists for both processors.
    - **Cons:** Requires dynamic page allocation and management.

### 2.2 UVA Mechanics and Challenges

- **Discrete GPU Setup:** Even with UVA, physical data movement between separate CPU and GPU memory systems is necessary.
- **On-Demand Management:** Handled by `cudaMallocManaged`. When the GPU accesses unmapped data, it generates a page fault.
- **Page Fault Handling Path:**
    - GPU sends the request to the CPU.
    - CPU OS and GPU driver handle the fault and send information back.
    - **Latency:** Can be in the range of tens of milliseconds.
- **IOMMU (Input-Output Memory Management Unit):**
    - Bridges I/O devices and memory, enabling Direct Memory Access (DMA) using virtual addresses (I/O virtualization).
    - In UVA, the CPU becomes actively involved in management alongside the IOMMU, which previously managed GPU memory as a peripheral device.
- **Communication Overhead:** Retrieving mappings from the CPU's IOMMU involves PCI-E communication and CPU interrupt service handlers, taking tens of microseconds.

### 2.3 Page Prefetching and Management

To reduce on-demand overhead, systems predict future addresses and bring them in advance.

- **Memory Prefetching Example:** In sequential operations like vector addition, arrays `a` and `b` are ideal candidates.
    
- **Code Example (Vector Addition):**
    
    ```
    // Sequential access pattern suitable for prefetching
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    c[idx] = a[idx] + b[idx];
    ```
    
- **Tree-Based Neighborhood Prefetching (NVIDIA):**
    
    - Addresses the non-sequential execution of thread blocks.
    - Divides a 2 MB chunk into two 1 MB sections, then into 64 KB nodes.
    - A page fault in a node triggers a fetch of the entire 64 KB node. If multiple nodes fault, it fetches the upper 1 MB node.
- **Memory Oversubscription:** Occurs when allocated virtual memory exceeds physical capacity, necessitating page eviction.

## Part 3: GPU Warp Scheduling (Module 7, Lesson 3)

### 3.1 Role and Position in the GPU Pipeline

The **warp scheduler** is a critical component of the GPU pipeline, situated after the fetch, decode, and register file access stages. Its primary function is to manage the execution of multiple warps that are concurrently ready for scheduling.

### 3.2 Comparison: CPU vs. GPU Instruction Scheduling

- **CPU Instruction Scheduler:**
    - Selects from available ready instructions, examining all instructions in the scheduler queue that cannot be divided into multiple cycles.
    - **Common Policies:**
        - **Oldest First:** Prioritizes instructions based on their age in the pipeline.
        - **Critical Instruction First:** Prioritizes instructions that unblock others, such as **load instructions** (to generate memory requests as soon as the address is ready) or instructions with a high number of **dependent instructions** (to make them ready sooner).
- **GPU Warp Scheduler:**
    - Implements an **in-order scheduler** per warp.
    - However, because it can select and execute from any warp that is currently ready, the final execution of the instruction stream appears as **out-of-order execution** across the entire SM.

### 3.3 Core Scheduling Policies: RR vs. GTO

- **Round-Robin (RR):**
    - Selects one instruction from each warp in a sequential, rotating fashion.
    - **Performance Impact:** If multiple warps reach a memory instruction, they generate memory requests almost simultaneously. This can lead to pipeline stalls because the latency cannot be effectively hidden if all warps are waiting for memory returns at the same time.
- **Greedy-Then-Oldest (GTO):**
    - Focuses on a single warp (the oldest one) and continues to schedule it until it encounters a **long latency event**.
    - **Trigger Events for Switching:** Cache misses, branches, or lengthy operations (e.g., tensor core operations).
    - **Performance Impact:** By prioritizing the oldest warp, it generates its memory requests early enough to potentially reduce total stall cycles, though the benefit is highly dependent on specific program characteristics.

### 3.4 Two-Level Scheduling for Power Efficiency

Probing the entire scheduler queue to determine which instructions are ready consumes significant power. To mitigate this, **two-level scheduling** is employed:

- **Warp Categorization:** Warps are split into two groups: **Pending warps** and **Active warps**.
- **Mechanism:** The scheduler only probes and selects instructions from the **active warps queue**, significantly reducing the search space and associated energy consumption.

### 3.5 Interaction Between Scheduling and Cache Behavior

Warp scheduling decisions directly influence the L1 cache's effective working set size and hit rate.

- **Cache Thrashing in Round-Robin:** In a scenario where the cache can only hold three blocks, if four warps (using RR) access four distinct addresses ($A1, A2, A3, A4$), the fourth access will evict the first. Under an LRU (Least Recently Used) policy, this results in constant misses for every block as they are continuously evicted before reuse.
- **GTO Cache Advantage:** Because GTO stays with one warp longer, it allows that warp to reuse its data in the cache before moving to the next, potentially resulting only in **compulsory misses**.
- **Working Set Size Formula:**
    - For **Round-Robin**: $Working Set = SS_W \times N$ (where $SS_W$ is the working set size of a single warp and $N$ is the total number of warps in the SM).
    - For **Limited/GTO Scheduling**: $Working Set = SS_W \times SW$ (where $SW$ is the number of currently schedulable warps).

### 3.6 Advanced Scheduling Policies

- **Cache Conscious Scheduling (Roger 12):**
    - Limits the number of schedulable warps to a small quantity to keep the working set within the cache capacity.
    - **Trade-off:** Reducing the number of active warps minimizes cache working set size but restricts **Warp Level Parallelism (WLP)**.
    - **Constraint:** Must ensure all warps eventually execute to satisfy thread synchronization requirements.
- **Prefetching-Aware Warp Scheduler:** Built on the two-level scheduler; it selects data from non-consecutive warps to enhance **bank-level parallelism**.
- **CTA-Aware Scheduling:** Takes into account the characteristics of **Cooperative Thread Arrays (CTAs)** to control which specific CTAs are prioritized for scheduling.
