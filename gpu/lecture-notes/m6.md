# GPU Architecture Technical Manual: Part 1 - Divergent Branches and Control Flow

## 1. Manual Objectives and Scope

This manual details the performance implications of divergent branches, mechanisms for handling control flow in SIMT (Single Instruction, Multiple Thread) architectures, and strategies for register file and memory optimization.

**Key Objectives:**

- Analyzing performance issues stemming from divergent branches.
- Evaluating hardware mechanisms for managing thread divergence.
- Identifying opportunities for GPU register file (RF) optimization.
- Differentiating between shared memory and register files.

## 2. Module 1: Handling Divergent Branches

GPU execution typically follows the **SPMD (Single Program Multiple Data)** model, where a warp (a group of threads) is expected to execute the same program path together.

### 2.1 Definition of Divergent Warps

Divergence occurs when threads within the same warp execute different parts of a program.

- **Divergent Branch:** A branch where not all threads in a warp follow the same path.
- **Divergent Warp:** A warp currently executing a divergent branch.
- **Active Mask:** A bitmask used to indicate which threads are active at a given point in the program (e.g., `1, 1, 1, 1` or `1, 1, 0, 0`).

### 2.2 Mechanism: If-Else Example

When an if-else statement is encountered, the hardware must manage two distinct paths. **Example Logic:**

```
if (threadID < 2) {
    // Basic Block 2 (BB2): Work 1
} else {
    // Basic Block 3 (BB3): Work 2
}
// Basic Block 4 (BB4): Reconvergence Point
```

- **Execution Flow:** Threads with ID < 2 execute BB2 while others are masked out. Then, threads with ID >= 2 execute BB3 while the first group is masked.
- **Reconvergence Point:** The location where all threads in the warp begin executing together again.
    - **Compiler Term:** Immediate Post Dominator.

### 2.3 If-Else Conversion: Predicated Execution

Divergent branches can be eliminated through **predicated execution**, where instructions are assigned a predicate value.

- **Operation:** An instruction only commits its results if its predicate value is true.
- **Pros:** Common in vector processors and modern CPUs for branches with small amounts of work.
- **Cons:** The "biggest downside" is that instructions for the non-executed path must still be fetched, which is inefficient if the work in the branches is large.

### 2.4 GPU Execution Flow and Detection

When a processor fetches an instruction, it computes the address of the next instruction.

- **Divergence Detection:** The hardware detects divergence by checking if all Program Counter (PC) values within a warp are identical.
- **The Address Problem:** If the hardware simply follows one path (e.g., BB2 then BB4), it might completely miss the alternative path (BB3) unless it has a mechanism to store the alternative address.

### 2.5 The SIMT Stack

To handle nested branches without losing alternative PC information, GPUs utilize a **SIMT Stack**.

- **Push Operation:** When divergence is detected at a branch, the processor pushes the alternative PC value and the reconvergence point onto the stack.
- **Nested Branches:** For complex control flow (e.g., a branch inside BB2), new alternative PCs and reconvergence points are pushed onto a second level of the stack, similar to handling nested function calls.
- **Pop Operation:**
    1. The processor fetches and executes instructions until the next PC address matches the **reconvergence point** stored at the top of the stack.
    2. Upon a match, instead of fetching the next PC, the processor **pops** the stack.
    3. It then begins fetching from the **alternative PC value** retrieved from the stack.

### 2.6 Identifying Reconvergence Points

The identification of these points is critical for SIMT support.

- **Compile-Time Analysis:** Typically done using a Control Flow Graph (CFG).
- **Hardware Implementation:** Hardware prefers to use a dedicated stack to detect divergence and store reconvergence points purely based on hardware information.
- **Compiler-Assisted Implementation:** In some cases, the compiler inserts markers or explicit stack operations to specify when to push and pop.
    - **Downside:** It is difficult to detect all divergent branch information at compilation time, often leading to over-claiming branches as divergent.

### 2.7 Micro-architectural Considerations: Large Warps

The width of a warp (e.g., 32, 64, or 128 threads) is a micro-architectural feature with specific trade-offs.

- **Benefits:** Larger warps allow one instruction fetch to generate many execution instructions, creating a wide execution machine.
- **Downsides:** The probability of divergence is significantly higher in larger warps.
- **Application Sensitivity:** Applications with **uniform** execution paths benefit from large warps, while those with frequent **divergent** branches suffer.

### 2.8 Dynamic Warp Formation

Because divergent warps result in low execution unit utilization (e.g., 50% if only half the warp is active), **Dynamic Warp Formation** is used to regroup threads.

- **Concept:** Regroup threads from different warps that are executing the same PC into a new warp.
- **Goal:** Increase execution width utilization.
- **Challenge (Register File Conflict):** Threads are typically positioned within a warp to minimize register file read/write conflicts. Regrouping them dynamically must preserve these conflict-avoidance patterns to prevent performance degradation.

# GPU Architecture Technical Manual: Part 2 - Register File and Memory Optimizations

## 3. Module 2: Register File (RF) Challenges and Observations

Modern Streaming Multiprocessors (SMs) manage a vast number of threads, each requiring its own set of registers. This architectural requirement creates significant scaling and performance hurdles.

### 3.1 Register File Scale and Constraints

- **Capacity:** Total register file size per SM can range from **64 KB to 256 KB**.
- **Latency and Bandwidth:** Large structures are inherently slower; however, in a GPU, the **RF read bandwidth must equal the execution bandwidth** to prevent bottlenecks.
- **Optimization Goals:** Reducing access latency through hierarchical structures and reducing physical size through resource sharing.

### 3.2 Key Micro-architectural Observations

Four primary observations regarding GPU register usage provide the foundation for optimization:

1. **Thread Activity Variance:** While GPUs follow the SPMD model, execution is **asynchronous**. Threads complete at different times, and registers held by finished threads are no longer required. Furthermore, divergent branches cause register usage patterns to vary dynamically at runtime.
2. **Register Liveness:** A register is **"live"** if its value will be used again and **"dead"** if the value is no longer needed.
    - **Example Code:**
        
        ```
        add r1, r2, 1  // r2 is read
        move r2, 10    // r2 is overwritten; old value is now 'dead'
        ```
        
    - In GPUs, the "dead" period between instructions for a specific thread can span **tens of warp instructions** because the scheduler rotates through many different warps.
3. **Instruction Proximity:** Conversely, if a scheduler places a dependent instruction (like the `add` above) immediately after the producer (`move`), the value might not need to be committed to the main RF at all.
4. **Read-Once Prevalence:** Approximately **70% of register values** are read only once before being overwritten. This suggests that many values do not strictly require long-term storage in a large register file.

### 3.3 Optimization Technique 1: Hierarchical Register Files

To balance capacity and speed, GPUs can implement a multi-level hierarchy.

- **Register File Cache:** An **L0 or L1 structure** provides fast, low-latency access to frequently used values.
- **Partitioned Register File:** This approach uses different storage technologies for different levels. For instance, **non-volatile memory** can be used for the large, slow portion of the RF to reduce energy consumption.

### 3.4 Optimization Technique 2: Register File Virtualization

This technique decouples the program's register namespace from the physical hardware.

- **Mechanism:** A small physical register file stores actual values, while a separate **namespace management** system tracks virtual registers.
- **Benefit:** This allows for aggressive **resource sharing**, as the hardware only allocates physical storage for registers that are actually in use (live).

## 4. Module 3: Unified Register File and Shared Memory

Resource constraints directly impact **occupancy**—the number of active blocks an SM can support.

### 4.1 Occupancy Constraints

- **The Problem:** Register file size and shared memory size are typically separate, fixed hardware limits.
- **Scenario A:** A block uses many registers but no shared memory; shared memory remains **underutilized**.
- **Scenario B:** A block uses high shared memory but few registers; the RF remains **underutilized**.

### 4.2 Comparative Analysis: RF vs. Software-Managed Cache

|Feature|Hardware Cache|Software-Managed Cache (Shared Mem)|Register File (RF)|
|:--|:--|:--|:--|
|**Management**|Hardware (LRU, etc.)|Programmer specified|Compiler/Static|
|**Addressing**|Requires Address Tags|Location-based (No Tags)|Location-based (No Tags)|
|**Ports**|Typically 1 read/write|Typically 1 read/write|Multiple (2+ source operands)|
|**Access Pattern**|Dynamic/Unknown|Known/Software defined|Known at static time|

### 4.3 Proposed Solution: Unified Structures

A unified structure allows the SM to dynamically allocate the same on-chip memory pool for either register files or shared memory.

- **Benefit:** This flexibility increases occupancy by making the **summation** of RF and shared memory the only limiting factor.
- **Requirements:**
    - **Extremely High Bandwidth:** Must support both register and memory access patterns simultaneously.
    - **Advanced Banking:** Requires complex banking structures to handle the varied port requirements and avoid conflicts.
