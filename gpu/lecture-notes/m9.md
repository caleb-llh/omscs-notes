## I. Multi-GPU Hardware Architecture

### 1. Evolution and Scaling Strategies

To meet increasing computing demands, GPU architectures have evolved beyond monolithic designs through three primary scaling stages:

- **Core Scaling:** Initial evolution focused on increasing the number of cores within a single chip.
- **Physical Integration:** To overcome cost and reliability challenges of monolithic approaches, scaling transitioned to connecting multiple GPU chips using **silicon interposers** or **multi-sockets**.
- **Board-Level Scaling:** Further expansion is achieved by connecting multiple GPU boards via multiple **PCI Express (PCIe)** connections.

### 2. Multi-GPU Connections and Packaging

Scaling beyond Moore’s Law is enabled through packaging-level integration to create larger logical GPUs.

- **Logical View:** From a programmer’s perspective, a multi-GPU module is still viewed as a collection of many **Streaming Multiprocessors (SMs)**.
- **Memory Architecture:** All memories are accessible via **global memory**, functioning as a shared memory program space.
- **High Bandwidth Memory (HBM):** 3D stacked memory, such as HBM, is utilized to meet high DRAM bandwidth requirements.
- **I/O Sharing:** The total system I/O bandwidth must be shared across the integrated modules.

### 3. Memory Management Challenges (NUMA)

Multi-GPU systems introduce **Non-Uniform Memory Access (NUMA)**, which creates variances in memory latency and bandwidth based on proximity.

- **Near Memory:** Memory local to the GPU where a thread block is executing.
- **Far Memory:** Memory located on a different GPU chip or module, characterized by higher latency and lower bandwidth.

#### 3.1 Thread Block Scheduling and Page Allocation

Two primary questions arise in NUMA environments: how to schedule thread blocks and how to allocate pages.

- **First-Touch Allocation:** A simple solution where a page is allocated in the "near memory" of the GPU where the requesting thread block is first scheduled.
- **Round-Robin Scheduling:** Thread blocks are distributed across available GPUs. For example, in a system with multiple GPUs, each GPU might receive one thread block (e.g., GPU 0 gets Thread Block 0, GPU 1 gets Thread Block 1).
- **Locality Optimization:** It is critical to schedule thread blocks on GPUs that provide near-memory access to the required data.
    - _Example:_ If Thread Block 5 requires Page 8, the scheduler should ideally place Thread Block 5 on the GPU where Page 8 is already located to avoid far-memory accesses.
- **Memory Migration:** As an alternative to locality-aware scheduling, memory pages can be migrated between GPUs if the frequency of far-memory accesses justifies the migration overhead.

### 4. Interconnect Technologies: NVLink and NVSwitch

**NVLink** is a proprietary high-speed communication protocol designed as an alternative to the PCIe bus for GPU-to-GPU communication.

- **Performance (4th Generation):**
    - **Bandwidth:** Provides **900 GB/s** per GPU.
    - **Links:** Features **18 links** per GPU.
    - **Comparison:** For context, an H100 GPU with HBM3 has **3 TB/s** of memory bandwidth; NVLink provides approximately one-fourth of that bandwidth for inter-GPU communication.
- **NVSwitch:** An introductory switch technology designed to provide point-to-point communication scalability as the number of GPUs increases.
- **Combined Impact:** The integration of NVSwitch and NVLink results in higher bandwidth and shorter latency across large GPU clusters.

### 5. Remote Direct Memory Access (RDMA)

Traditional communication (e.g., TCP/IP) requires traversing the network software stack and consumes CPU resources, often being limited by the CPU's busy status.

- **RDMA Definition:** Remote Direct Memory Access enables direct memory-to-memory communication without CPU intervention.
- **Host-Bypass Technology:** RDMA allows data to move directly between memory buffers (e.g., from Buffer 4 to Buffer 2) without passing through the host CPUs.
- **GPUDirect RDMA:**
    - Allows direct buffer-to-buffer communication via the network interface.
    - Leverages NVLink and NVSwitch for high-demand communication that the PCIe bus cannot satisfy.
    - Enables communication between GPUs and even third-party devices.

## II. Concurrency on GPUs

### 1. Utilization and Workload Management

Increasing GPU utilization is critical, especially in data center environments.

- **Traditional Data Centers:** Typically handle **multi-tenant jobs**.
- **Modern AI Workloads (e.g., LLMs):** Often consume all resources of a single GPU for a **single tenant**.
- **Performance Bottlenecks:** AI workload performance can be limited by the slowest task in the pipeline.

### 2. Concurrency Support Mechanisms

The architecture includes SMs (with L1/L2 caches) and DRAM, supporting three primary levels of concurrency:

|Mechanism|Description|Isolation Level|
|:--|:--|:--|
|**Streams**|Multiple streams executed concurrently; must be independent; typically from a single client.|Software/Logical|
|**Multi-Process Service (MPS)**|Different clients use different SMs while sharing other resources.|Spatial (Partial)|
|**Multi-Instance GPU (MIG)**|GPUs are physically separated for each client.|Hardware/Full|

#### 2.1 Multi-Instance GPU (MIG)

MIG is designed for high-efficiency utilization of large GPUs in cloud and multi-tenant scenarios.

- **Resource Isolation:** Provides strict isolation for:
    - GPU cores
    - Memory and L2 cache ports
    - DRAM memory bandwidth
    - On-chip crossbar ports
- **Quality of Service (QoS):** Ensures predictable performance for multiple users.
- **Capacity:** An NVIDIA A100 can host up to **7 instances**.

#### 2.2 Multi-Process Service (MPS)

Available since the V100, MPS provides spatial partitioning.

- **Resource Sharing:** Unlike MIG, it does not strictly partition resources, which can lead to unpredictable performance.
- **Use Case:** Facilitates the porting of MPI (Message Passing Interface) jobs to CUDA. It may offer better utilization than MIG when strict QoS is not a priority.

## III. Stream-Based Programming and Optimization

### 1. Functional Overview of Streams

Streams enable the **concurrent execution of multiple kernels** on a single GPU.

- **Concurrency:** Multiple CPU threads can submit kernels to different streams simultaneously.
- **Asynchronous Operations:** The architecture supports `cudaMemcpyAsync`, which allows for the **overlap of communication and computation** across different streams.
- **Portability:** Stream-based programming provides a straightforward path for porting OpenMP programs to the GPU environment.

### 2. CUDA Graphs

While streams facilitate concurrency, scheduling among multiple streams can introduce **performance overhead** due to the necessity of checking dependencies.

- **Mechanism:** CUDA Graphs are used to construct a **dependency chain** before a kernel is launched.
- **Benefit:** This pre-construction of the execution flow helps overcome the scheduling overhead associated with standard stream management.

## IV. Multi-GPU Programming Workflow

The programming model for multi-GPU systems requires explicit management of memory and device context.

### 1. Architectural Workflow Example

Based on the provided Gaussian PDF execution model, the technical workflow for multi-GPU orchestration is as follows:

1. **Memory Allocation:** Distinct memory objects must be allocated for **each individual GPU**.
2. **Device Selection:** Use `cudaSetDevice` from the host side to specify which physical GPU the subsequent commands should target.
3. **Kernel Invocation:** Call the specific kernel (e.g., `gaussian_pdf`) for each GPU separately.
4. **Synchronization:** Implement a barrier to **wait until all devices finish** their respective tasks before proceeding.
5. **Hybrid Approach:** Note that **multi-streams** can be utilized within each GPU in a multi-GPU setup to further increase concurrency.

## V. GPU Support for Multi-Tenant Computing: Comparative Analysis

The following table synthesizes the partitioning and isolation characteristics of different GPU concurrency techniques:

|Feature|Streams|Multi-Process Service (MPS)|Multi-Instance GPU (MIG)|
|:--|:--|:--|:--|
|**Partitioning Type**|Logical/Software|Spatial (Partial)|Hardware (Physical)|
|**SM Isolation**|None (Shared)|Isolated per client|Fully Isolated|
|**Memory Bandwidth Isolation**|Shared|Shared|Isolated|
|**Performance QoS**|Low|Medium (Variable)|**High (Predictable)**|
|**Partitioning Time**|Runtime|Runtime|Configuration-time|

### 1. Strategic Selection Criteria

- **MIG:** Preferred for **cloud service providers** and multi-tenant scenarios where strict **Quality of Service (QoS)** and hardware-level isolation are required.
- **MPS:** Recommended for porting **MPI jobs** to CUDA or when maximum utilization is desired and performance variability (due to lack of strict partitioning) is acceptable.
- **Streams:** Ideal for a **single client** managing multiple independent tasks to maximize the occupancy of a single logical GPU instance.
