## 1. Multi-threaded Architecture

### 1.1 Fundamental GPU Recap

**GPUs** are designed with a high density of **cores** to support massively parallel execution. Each individual core features **multithreading** capabilities, which allows for the execution of **warps** (or wavefronts) within that core. To support these operations, every core is equipped with hardware **caches** and **shared memory**.

### 1.2 Comparative Execution Paradigms

The efficiency of a processor is often defined by how it handles instruction flow through its pipeline, specifically during long-latency events like cache misses.

- **In-Order Processors:** In a standard five-stage pipeline, an instruction that suffers a **cache miss** will cause all subsequent instructions to wait until the first completes, even if the following instructions have no data dependencies.
- **Out-of-Order (OoO) Processors:** These processors attempt to mitigate stalls by executing independent instructions (e.g., instructions 2 and 3) while a prior instruction (instruction 1) is waiting for a memory request.
- **GPU Multithreading:** Unlike the complex dependency checking of OoO, a GPU simply **switches to another thread** whenever a stall occurs, or even regardless of whether a miss has occurred. Because instructions from different threads are inherently independent, the processor can execute instructions from multiple threads and generate multiple **concurrent memory requests**.

### 1.3 Benefits of Multithreading and Latency Hiding

The primary advantage of GPU multithreading is its ability to **hide processor stall time**.

- **Stall Contributors:** Latency is typically introduced by **cache misses**, **branch instructions**, or long-latency **ALU instructions**.
- **Latency Mitigation:** While CPUs use complex hardware like OoO execution, large cache systems, and Instruction Level Parallelism (ILP), GPUs rely on a **greater number of threads** to hide longer memory latencies.

### 1.4 Hardware Support and Resource Requirements

Supporting high-level multithreading requires specific hardware extensions in the GPU's front-end and storage.

- **Front-end Extension:** To manage multiple warps, the hardware must maintain multiple **Program Counter (PC) values**. Every warp requires its own PC register; for example, supporting four concurrent warps requires four PC registers.
- **Register File (RF):** A significantly large register file is mandatory because **context switching** in a GPU is instantaneous. It is achieved by simply switching a pointer among the various PC registers and register file sets.
- **CPU Comparison:** In contrast, CPU **context switches** involve storing the current thread’s contents (PC, registers) into memory and loading the new thread’s contents from memory. This process incurs substantial **performance overhead**.

### 1.5 Occupancy and Scaling Calculations

The total size of the register file is determined by the formula: **Total RF Size = K (Registers per Thread) × Number of Threads**.

- **PC Relation:** The number of threads a Streaming Multiprocessor (SM) can execute is directly tied to the number of available PC registers. If hardware provides five PC registers, and each warp consists of 32 threads, it can support 160 threads ($5 \times 32$).
- **Execution Example:** If an SM can execute **256 threads** (with 32 threads per warp), it requires **8 PC registers** ($256 / 32$).
- **Instruction Fetch Example:** If a program contains 10 instructions and is executed by 8 warps (8 PCs), the SM must fetch an instruction **80 times** ($10 \times 8$).

## 2. Bank Conflicts and Resource Bandwidth

### 2.1 Register File Access in SIMT

In the Single Instruction, Multiple Thread (SIMT) model, **registers are per-thread**. For a GPU with an execution width of eight, an instruction requiring two source operands and one write operand must supply **24 values simultaneously** ($8 \text{ threads} \times 3 \text{ operands}$).

### 2.2 Port vs. Bank Architectures

To provide the necessary bandwidth for these accesses, hardware designers choose between increasing ports or using banks.

- **Ports:** A port is the physical hardware interface for data access. High port counts (e.g., 8 read and 4 write ports per register element) require extensive wiring and consume significant die space.
- **Banks:** A bank is a **partition or group** of the register file. Multiple banks can be accessed simultaneously, allowing for high bandwidth with fewer ports per bank. A four-bank version is much smaller than a high-port version because it reduces wiring requirements.

### 2.3 Bank Conflict Mechanics

A **bank conflict** occurs when multiple threads within a warp attempt to access different data stored in the **same bank** simultaneously.

- **Scenario 1 (No Conflict):** If Thread 1, 2, 3, and 4 each read register R1 from different banks, the values can be provided in a single cycle.
- **Scenario 2 (Conflict):** If a single thread (or multiple threads mapping to the same bank) needs to read R1, R2, R3, and R4 simultaneously, a four-bank system with only two read ports will take **multiple cycles** to provide the data.

### 2.4 Variable Register Counts and Optimization

The number of registers assigned to a thread can change how threads map to banks.

- **Case 1 (High Register Count):** If each thread is assigned four registers, reading R1 and R2 across multiple threads may avoid conflicts if the hardware layout ensures they land in different banks.
- **Case 2 (Low Register Count):** If each thread is assigned only two registers, threads 1 and 2 might end up in the same bank, causing a conflict during a standard `R3 = R1 + R2` operation.
- **Compile-Time Solutions:** Compilers can perform **static analysis** (analysis before the code runs, independent of program input) to change instruction order and remove bank conflicts.

### 2.5 Scoreboarding and Buffering

Modern GPU pipelines are expanded beyond five stages to handle the complexities of register access.

- **Buffers:** Read values are stored in a **buffer** until they are ready for execution.
- **Scoreboarding:** The GPU uses a **scoreboard** for dynamic instruction scheduling. It checks if all source operands for a warp are ready by monitoring **ready bits** in the buffer.
- **Scheduling Policy:** Once all operands (e.g., Warp 1, Src 1 and Src 2) are ready, the scoreboard selects a warp to send to the execution unit, often using an **oldest-first** policy.

### 2.6 Shared Memory Bank Conflicts

**Shared memory** (on-chip scratchpad memory) also uses banks to provide high bandwidth. Conflicts occur based on the memory address calculation.

**Example of Conflict-Inducing Code:**

```
// Shared memory indexing that causes conflicts
shared_input[threadIdx.x * 4];
```

In this code, if each bank is 4 bytes wide, Thread 1 accesses address 4, Thread 2 accesses address 8, and Thread 3 accesses address 12. Because these addresses all map to the **same bank**, the threads generate bank conflicts, requiring the software structure to be redesigned for efficiency.

## 3. GPU Architecture Pipeline

### 3.1 Pipeline Stages and Warp Scheduling

The GPU pipeline is designed to manage multiple warps across various stages of execution, extending beyond a basic five-stage model to accommodate multithreading and complex register access.

- **Warp Tracking:** The pipeline maintains **PC (Program Counter) values** for each warp to track their current instruction.
- **Warp Identification:** Warps are grouped by their origin block and thread range. For example:
    - **Warp 1:** Block 1, threads 1-4.
    - **Warp 2:** Block 1, threads 5-8.
    - **Warp 3:** Block 2, threads 1-4.
    - **Warp 4:** Block 2, threads 5-8.

### 3.2 Instruction Fetch and Decode

- **Fetch:** The front end fetches a single instruction (e.g., at address `0x8000`) for an **entire warp**.
- **Decode:** Instructions are sent to the decode stage. If an instruction contains a **constant or immediate value**, that value is broadcasted to the scoreboard buffers for all threads in the warp.
- **Special Registers:** Specific registers are reserved for metadata.
    - `tid.x`: Stores the **thread ID** within a block.
    - `ctaid.x`: Stores the **block ID** within a grid.

### 3.3 Scoreboarding and Execution Logic

The scoreboard acts as a gatekeeper for the execution stage.

- **Operand Readiness:** The processor accesses the register file for source registers (e.g., `R1`) for all threads in the warp (e.g., threads 1-4). Once these values are read and sent to the scoreboard, the **ready bits** are set.
- **Warp Selection:** When the scoreboard confirms all source operands for a warp are ready, the scheduler selects that warp to move to the **execution stage**.
- **Write Back:** After performing computations (e.g., additions or subtractions), the final results are written back to the register file at the **write-back stage**, updating the values for the specific threads in that warp.

### 3.4 Mask Bits and SIMT Lane Management

A **lane** represents a single ALU execution path.

- **Active vs. Inactive:** Not all threads in a warp necessarily perform work. **Mask bits** identify which threads (lanes) are active.
- **Execution Behavior:**
    - **Active Threads:** Perform the actual computation.
    - **Inactive Threads:** Do not perform any work during that cycle.
- **Example:** A mask bit of `1111` means all four lanes are active; a mask bit of `1110` means only the first three lanes will execute the instruction.

## 4. Global Memory Coalescing

### 4.1 Global Memory Request Generation

In GPU architecture, a single memory instruction can trigger a massive number of requests to DRAM.

- **Warp-Level Scaling:** Assuming a warp size of 32, one warp can generate up to **32 distinct memory requests**.
- **System-Wide Scaling:** If a GPU has 32 Streaming Multiprocessors (SMs), each executing one warp, they can generate **1,024 requests in a single cycle** ($32 \times 32$).
- **Bandwidth Saturation:** At 64 bytes per request and a 1 GHz clock speed, this requires **64 terabytes per second** of memory bandwidth, making it easy to saturate the memory system.

### 4.2 Memory Technology: SRAM, DRAM, and HBM

- **SRAM (Static RAM):** Composed of six transistors per bit; used for **caches** due to speed.
- **DRAM (Dynamic RAM):** Composed of one transistor per bit; provides large capacity but is limited by communication **pins**.
- **HBM (High Bandwidth Memory):** Overcomes DRAM pin limitations by **stacking DRAM** for higher density and using a **silicon interposer** to connect memory to the GPU. This keeps communication within the same package, significantly increasing bandwidth.

### 4.3 Memory Coalescing Mechanics

Coalescing is a technique that combines multiple memory requests into fewer, more efficient requests.

- **Coalesced Access:** When thread registers (e.g., `R1`) contain **sequential memory addresses**, the hardware combines these into a single request (e.g., one 64-byte request for addresses 0-63).
- **Uncoalesced Access:** When thread registers (e.g., `R2`) contain addresses that are **strided** (e.g., 128 bytes apart), each request must be sent separately. This significantly degrades performance.

### 4.4 Technical Comparison of Access Patterns

|Feature|Coalesced Access|Uncoalesced Access|
|:--|:--|:--|
|**Address Pattern**|Consecutive/Sequential|Strided/Disjoint|
|**Request Count**|Single or few efficient requests|Up to 32 separate requests per warp|
|**Performance**|Optimal; reduces bandwidth saturation|Poor; can saturate bandwidth quickly|

**Code Example of Coalescing Impact:**

```
// Potential Coalesced Access (Sequential)
Ld.global R1; // If R1 values are 0, 4, 8, 12...

// Potential Uncoalesced Access (Strided)
Ld.global R2; // If R2 values are 0, 128, 256, 384...
```
