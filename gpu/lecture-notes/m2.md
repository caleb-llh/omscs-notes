#### 2. Parallel Application Development Lifecycle

To transform a serial program into a parallel application, the workload must be divided, data structures adjusted for parallel execution, and tasks distributed across multiple processors.

**The Four-Step Engineering Process:**

1. **Step 1: Discover Concurrency:** Identify specific opportunities for parallelism within the problem domain. This serves as the foundational starting point.
2. **Step 2: Structuring the Algorithm:** Organize the code to effectively exploit the identified parallelism. Proper structuring is essential for harnessing parallel power.
3. **Step 3: Implementation:** Select the appropriate programming environment, languages, and tools for the structured algorithm.
4. **Step 4: Execution and Optimization:** Execute the code on a parallel system and fine-tune it to reach peak performance.

#### 3. Formal Parallel Programming Patterns

Five distinct patterns dominate the landscape of parallel programming:

##### 3.1 Master/Worker Pattern

- **Classification:** Single Program, Multiple Data (SPMD).
- **Architecture:**
    - **Master Process/Thread:** Manages a pool of worker processes/threads and a shared task queue.
    - **Worker Threads:** Concurrently dequeue and execute tasks from the shared queue.
- **Primary Utility:** Ideal for workloads that can be decomposed into smaller, **independent pieces**.

##### 3.2 Loop Parallelism Pattern

- **Mechanism:** Repetitive iterations within loops are executed in parallel.
- **Key Distinction:** Unlike the Master/Worker pattern, tasks inside the loops are typically identical.
- **Candidate Profile:** Loops where each iteration is independent of others.

##### 3.3 SPMD (Single Program, Multiple Data) Pattern

- **Mechanism:** All processing elements execute the exact same program simultaneously, but each element operates on its own unique set of data.
- **Prevalence:** This is a popular choice specifically for **GPU programming**.

##### 3.4 Fork/Join Pattern

- **Mechanism:** Combines serial and parallel processing phases.
- **Workflow:** A parent task creates (forks) a new task, then waits (joins) for the task's completion before resuming the next serial segment.
- **Utility:** Frequently used in programs featuring a single entry point.

##### 3.5 Pipeline Pattern

- **Analogy:** Resembles a CPU pipeline where processors operate on different stages of a task.
- **Data Profile:** Optimized for processing continuous data streams.
- **Workflow Examples:**
    - **Signal Processing**.
    - **Graphics Pipelines**.
    - **Compression Workflows:** Sequentially handling decompression, work, and compression.
- **Parallelism:** Parallelism is achieved by operating on different pieces of work at different stages simultaneously.

#### 4. Shared vs. Distributed Memory Models

##### 4.1 Shared Memory Systems

- **Definition:** All processors have access to a common memory space.
- **Interaction:** A processor (e.g., Processor 1) updates a value, and another (e.g., Processor 2) observes that update by reading directly from the shared memory.

##### 4.2 Distributed Memory Systems

- **Definition:** Each processor possesses its own private, isolated memory space.
- **Communication:** To access data in a different memory space, processors must explicitly exchange messages.
- **Example Operation:**
    - Processor 2 requires data located at the same memory address across different systems (Processor 1 and Processor 3).
    - Processor 2 sends a message request; Processor 1 sends its data (e.g., 0) and Processor 3 sends its data (e.g., 200).
    - Processor 2 performs the computation (e.g., addition) using these updated values and writes the result back to its local memory.

#### 5. OpenMP Architecture and Execution

##### 5.1 Overview

- **Definition:** An open standard for parallel programming on shared memory systems.
- **Components:** A collection of **compiler directives**, **library routines**, and **environment variables**.

##### 5.2 Key Parallel Programming Tasks

- Parallelization.
- Specifying thread counts.
- Scheduling.
- Data sharing.
- Synchronization.

##### 5.3 Thread Theory

- **Definition:** An entity that executes a sequence of instructions.
- **Resources:** Every thread maintains its own **register** and **stack memory**.
- **Hardware Relation:** A thread is a software concept; it is not equivalent to a core. One CPU core can execute one or multiple threads.
- **Note:** The definition of a thread differs between CPU and GPU architectures.

##### 5.4 Data Decomposition

- **Process:** Splitting an array across multiple cores.
- **Example:** A 10-element array is split in half; 5 elements are sent to Core 1 and 5 elements to Core 2 for parallel execution.

#### 6. Implementation Case Study: Vector Sum

##### 6.1 Serial Implementation

```
// Serial Vector Sum
for (int i = 0; i < 10; i++) {
    sum += array[i];
}
```

_Iterates 0 to 10 to compute the sum of elements._

##### 6.2 Manual Parallelization (Low-Level)

Requires manual handling using APIs like **p-threads**, including:

- Manual thread creation.
- Thread joining.
- Manual Mutex operations for data integrity.
- Merging partial sums into a final total (a **reduction operation**).

##### 6.3 OpenMP Implementation

```
// OpenMP Vector Sum
#pragma omp parallel for reduction(+:sum)
for (int i = 0; i < 10; i++) {
    sum += array[i];
}
```

- **`#pragma omp parallel for`:** Invokes the loop parallelism pattern and generates threads automatically.
- **`reduction(+:sum)`:** A compiler directive that handles the creation and joining of threads via the runtime library.
- **Mechanism:** The compiler replaces directives with library calls, which manage partial sum calculation and the final merge.

##### 6.4 Synchronization and Mutexes

- **Critical Section:** A segment of code where a shared variable (like `total sum`) is updated.
- **Mutex (Mutual Exclusion):** Ensures only one thread can access a critical section at a time.
- **Workflow:**
    1. **Lock (Acquire):** A thread acquires a mutex to enter the critical section.
    2. **Unlock (Release):** The thread releases the mutex after the update, allowing other threads to access it.
- **Risk:** Without a mutex, two threads updating a shared `sum` simultaneously might result in incorrect totals (e.g., 100 or 200 instead of 300) due to a data race.

#### 7. Thread Management and Resource Allocation

Control over the execution environment is central to optimizing OpenMP applications.

##### 7.1 Specifying Thread Counts

- **Environment Variables:** The `OMP_NUM_THREADS` variable is used to define the number of threads for parallel regions.
- **Default Behavior:** If not explicitly set, the number of threads typically defaults to the level of hardware parallelism available (e.g., the number of CPU cores).
- **Function Calls:** The `omp_set_num_threads()` function allows programmers to define thread counts directly within the source code.

##### 7.2 The Scheduling Challenge

Scheduling becomes complex when thread counts do not align perfectly with hardware resources.

- **Scenario:** Distributing 1,000,000 elements among 5 threads on only 2 cores.
- **Issue:** If one core handles three threads and the other handles two, the core with three threads has significantly fewer resources per thread, leading to uneven progress.

---

#### 8. OpenMP Scheduling Strategies

To mitigate uneven progress, OpenMP provides several scheduling options.

- **Static Scheduling (Option 1):** Allocates consistent, fixed chunks of work (e.g., 200k elements) to each thread. This assumes uniform progress across all threads.
- **Dynamic Scheduling (Option 2):** Allocates a minimal unit (1 element) to each thread. Threads request more work upon completion, ensuring high resource utilization but potentially high overhead.
- **Dynamic Scheduling (Option 3):** Allocates larger initial chunks (e.g., 1,000 elements) to balance the cost of requesting work with the benefit of dynamic load balancing.
- **Guided Scheduling (Option 4):** Features variable chunk sizes that decrease over time (e.g., starting at 1,000 and dropping to 100). This adapts to runtime conditions; faster threads receive larger initial chunks while slower threads receive smaller ones later.

---

#### 9. Data Sharing and Synchronization

##### 9.1 Data Sharing Policies

Programmers must explicitly distinguish between different data types to ensure correctness:

- **Private Data:** Variables unique to a thread, such as partial sums.
- **Shared Data:** Variables accessible by all threads, such as the final overall sum.

##### 9.2 Synchronization Constructs

- **Barrier (`#pragma omp barrier`):** A synchronization point where all threads must arrive before any can proceed. This is essential for tasks with dependencies, such as sorting or multi-stage updates.
- **Critical Section (`#pragma omp critical [(name)]`):** Ensures a block of code is executed by only one thread at a time. It prevents data races during shared variable updates, like incrementing a counter.
- **Atomic (`#pragma omp atomic`):** Guarantees a specific operation completes entirely or not at all.
    - **Logic:** A counter increment involves three steps: **Load**, **Add**, and **Store**. Atomic ensures these three steps occur as a single unit, preventing other threads from intervening.

---

#### 10. Advanced Directives: Parallel Sections

Parallelism can be applied to non-looping code blocks using the `sections` directive.

```
#pragma omp parallel sections
{
    #pragma omp section
    { work1(); }
    #pragma omp section
    { work2(); }
}
```

_Work 1 and Work 2 execute in parallel._

##### 10.1 Specialized Section Constructs

- **Ordered:** Ensures threads execute in ascending order. Without it, execution order is non-deterministic.
- **Single:** Ensures only one thread (any thread) executes the task. This is commonly used for initialization tasks to avoid redundant processing.

---

#### 11. Distributed Memory and MPI (Message Passing Interface)

##### 11.1 The Necessity of MPI

GPU programming (specifically CUDA) is complex because it blends shared and distributed memory concepts. While some regions are shared, others are isolated and require explicit communication.

##### 11.2 Core MPI Concepts

MPI is a communication model that allows isolated processes to exchange data.

- **Point-to-Point Communication:**
    - `MPI_send()`: Process 0 sends data to Process 1.
    - `MPI_recv()`: Process 1 receives data from Process 0.
- **Collective Communication:** `MPI_Bcast()` (Broadcast) sends data from one process to all others in the system.

##### 11.3 Stencil Operations Case Study

Stencil operations, common in High-Performance Computing (HPC), involve calculating a value based on a neighborhood of data (e.g., an average of four neighbors).

- **The Problem:** In a distributed system, a process may need data residing in another process's memory to complete a calculation at the boundary.
- **The Solution:** **Boundary Information Communication**. Processes must send "gray" boundary data to their neighbors via messages to ensure every process has the neighborhood data required for its local stencil calculations.

### GPU Computing Technical Manual: Part 3 - Advanced OpenMP Directives, Sync Logic, and MPI Communication

#### 12. Thread vs. Core Architecture Concepts

- **Thread Definition:** An entity executing a sequence of instructions, characterized by its own **register** and **stack memory**.
- **Decoupling from Hardware:** A thread is a software concept and is **not equivalent to a core**.
    - One CPU core can execute a single thread or multiple threads concurrently.
    - **Architecture Note:** The semantic definition of a thread differs between CPU and GPU contexts.
- **Data Decomposition Model:** In a 10-element array split across two cores, each core handles five elements. While modern CPUs execute multiple threads per core, standard architectural models often assume a 1:1 core-to-thread ratio for simplification.

---

#### 13. Deep Dive: Shared Memory Reduction and Mutex Logic

Merging parallel results into a singular output requires specific memory management.

##### 13.1 Reduction Operation Workflow

1. Each thread computes its own **partial sum**.
2. One thread updates the **total sum** variable in shared memory.
3. Subsequent threads read the updated total, add their partial sum, and write the new value back.

##### 13.2 Mutex (Mutual Exclusion) and Critical Sections

- **The Data Race Risk:** Without protection, if two threads attempt to update a shared `sum` simultaneously, the result might reflect only one of the additions (e.g., resulting in 100 or 200 instead of the correct 300).
- **Critical Section:** A segment of code that updates shared data and must only be accessed by one thread at a time.
- **Protocol:**
    - **Locking:** Acquiring a Mutex before entering the critical section.
    - **Unlocking:** Releasing the Mutex after the update to allow other thread access.

---

#### 14. OpenMP Directive Syntax and Advanced Constructs

OpenMP uses `#pragma omp directive [clause [clause] ...]` semantics to communicate with the compiler.

##### 14.1 Atomic Operations

- **Directive:** `#pragma omp atomic`
- **Logic:** Guarantees that an operation (like incrementing a counter) is performed as a single unit.
- **Internal Mechanics:** An increment involves **Load**, **Add**, and **Store**; `atomic` ensures all three occur together or not at all, preventing data races.

##### 14.2 Parallel Sections and Specialized Constructs

Used for parallelizing non-loop code blocks.

- **Standard Sections Code:**
    
    ```
    #pragma omp parallel sections
    {
        #pragma omp section
        { work1(); }
        #pragma omp section
        { work2(); }
    }
    ```
    
    _Work 1 and Work 2 execute in parallel._
    
- **Ordered Construct:** Ensures threads execute and print in **ascending order**.
    
    ```
    #pragma omp parallel for ordered
    for (...) {
        #pragma omp ordered
        printf(...); // Prints in sequence (1, 2, 3...)
    }
    ```
    
    _Without `ordered`, output is non-deterministic._
    
- **Single Construct:** Ensures only **one thread** (regardless of which) executes the task, often used for initialization.
    
    ```
    #pragma omp parallel
    {
        #pragma omp single
        printf("Initialization task..."); // Prints only once
    }
    ```

#### 15. The MPI (Message Passing Interface) Model

Distributed memory systems require explicit communication protocols because each process can only access its own memory.

##### 15.1 Point-to-Point and Collective Communication

- **`MPI_send()` / `MPI_recv()`:** Used for direct data transfer between two processes (e.g., Process 0 sends an integer to Process 1).
- **`MPI_Bcast()` (Broadcasting):** A collective communication function that sends data from a single source process to **all** other processes in the system.

##### 15.2 Stencil Operations and Boundary Data

- **Stencil Logic:** Computing a value (e.g., `C`) by averaging its four immediate neighbors.
- **Distributed Challenge:** If `C` is at a memory boundary, some of its neighbors might reside in a different process's memory.
- **Boundary Information Communication:** Essential for stencil operations; "gray" boundary data must be sent across processes via messages to satisfy local neighborhood dependencies.

---

#### 16. Structural Synthesis: Why CUDA Blends Models

To address your query regarding CUDA's hybrid nature, the sources specify that CUDA programming is complex because it incorporates both paradigms:

1. **Shared Memory Aspect:** Certain memory regions are accessible to all cores.
2. **Distributed Memory Aspect:** Other memory regions are "invisible" to certain cores and require **explicit communications** similar to the MPI model to move data between isolated spaces.
