#### I. Module 3 Lesson 1: Introduction to CUDA Programming

**1.1 Primary Learning Objectives**

- Development of kernel code for **Vector Addition** operations.
- Mastery of basic CUDA programming terminologies.
- Analysis and explanation of Vector Addition code structures.

**1.2 Fundamental Paradigms: SPMD**

- **Single Program Multiple Data (SPMD):** GPUs execute code concurrently following this paradigm.
- **Concurrent Execution:** Multiple threads execute the same program in parallel.
- **Ordering:** Threads operate **without any specific ordering** (e.g., four threads printing messages may appear in any sequence).
- **Data Differentiation:** To ensure threads operate on different data despite running the same program, unique identifiers must be assigned.

**1.3 Thread Identification and Indexing**

- **Built-in Variables:** Identifiers are represented by built-in variables that map to spatial coordinates:
    - `threadIdx.x`, `threadIdx.y`, `threadIdx.z`: Represent the thread index within a block.
    - `blockIdx.x`, `blockIdx.y`: Represent the block index within a grid.
- **Dimensionality:** Indices can be two-dimensional or three-dimensional to facilitate mapping to physical objects like images or 3D models.
- **Global Index Calculation:** To access elements in a linear vector (e.g., 16 elements divided into 4 blocks of 4 threads each), the following pattern is used:
    - `idx = blockIdx.x * blockDim.x + threadIdx.x`.

**1.4 CUDA Execution Hierarchy**

- **Thread:** The basic unit of execution.
- **Warp:** A microarchitecture concept representing a group of threads executed together. While historically managed by hardware, modern CUDA exposes the warp concept to programmers.
- **Block (CUDA Block):**
    - A group of threads that form a block and are executed concurrently.
    - **Memory Scope:** Threads within a block have shared memory access scopes distinct from other blocks.
    - **SM Mapping:** For simplicity, assume each block is executed on one **Streaming Multiprocessor (SM)**.
    - **Execution Order:** There is **no guaranteed order** of execution among blocks.
- **Grid:** Consists of multiple blocks. The total number of threads in a grid is the product of `gridDim` and `blockDim`.

**1.5 Memory Spaces and Scopes**

- **Shared Memory (`__shared__`):**
    - Functions as a software-managed scratchpad memory.
    - Located **on-chip** for faster access compared to global memory.
    - **Scope:** Accessible **only within a single CUDA block**.
    - **Communication:** Data sharing between blocks requires explicit communication as shared memory is limited to its specific SM.
- **Global Memory:**
    - Resides outside the chip (device memory).
    - **Scope:** Accessible by all blocks within a kernel.
- **Local Memory:**
    - **Scope:** Private to an individual CUDA thread.
- **Constant Memory:**
    - Very small, read-only memory for data like mathematical constants (e.g., pi).
- **Texture Memory:**
    - Read-only memory structured for two-dimensional accesses, primarily used in 3D graphics.

**1.6 Synchronization and Execution Patterns**

- **Thread Synchronization:**
    - Threads progress at different speeds and in arbitrary orders.
    - `__syncthreads()`: Synchronizes all threads within a block until they reach that specific point in the code.
- **Interblock Synchronization:** Achieved only through separate kernel launches.
- **Bulk Synchronous Parallel (BSP) Pattern:**
    
    1. **Load:** Move data from global memory to shared memory.
    2. **Compute:** Perform operations using data in shared memory.
    3. **Store:** Write results from shared memory back to global memory.
    
    - `__syncthreads()` is vital between the Load and Compute stages to resolve data dependencies.

**1.7 Kernel Launch Syntax**

- **Execution Configuration:** Initiated from host code using three angle brackets `<<<numBlocks, threadsPerBlock>>>`.
- **Arguments:** Kernels accept standard arguments (e.g., `d_A, d_B, d_C, numElements`).
- **Multi-Kernel Workflow:** Different kernels can be launched for specific tasks (e.g., `sort`, `add`, `store`).
    - Example: A `sort` kernel might use only one thread, followed by an `add` kernel using `gridDim * blockDim` threads.

#### II. Module 3 Lesson 2: Occupancy

**2.1 Definition and Importance**

- **Occupancy:** Refers to how many CUDA blocks can reside and execute on a single Streaming Multiprocessor (SM).
- **Hardware Utilization:** Higher occupancy generally indicates higher parallelism and better hardware resource utilization.
- **Performance:** Case studies show that increasing threads per SM (even with smaller block sizes) can lead to higher performance (e.g., 32 threads/SM vs. 24 threads/SM).

**2.2 Constraints on Occupancy** The number of blocks per SM is determined by the hardware's minimum constraint among three factors:

1. **Number of Registers:**
    - The compiler determines the number of architectural registers required per thread.
    - Unlike CPUs with fixed ISA register counts, CUDA register requirements vary by code.
2. **Shared Memory Size:**
    - Determined by the code (e.g., `__shared__ int sharedMemory` allocates 4,000 bytes).
3. **Number of Threads:**
    - Determined by the host at kernel launch (e.g., `threadsPerBlock`).

**2.3 Occupancy Calculation Example**

- **Hardware Resources:** 256 thread capacity, 64 KB register file, 32 KB shared memory.
- **Software Requirements:** 32 threads per block, 64 registers per thread, 2 KB shared memory per block.
- **Constraint Analysis:**
    - **Threads Constraint:** $256 / 32 = 8$ blocks.
    - **Registers Constraint:** $64 \text{ KB} / (32 \text{ threads} \times 64 \text{ registers}) = 32$ blocks.
    - **Shared Memory Constraint:** $32 \text{ KB} / 2 \text{ KB} = 16$ blocks.
- **Final Occupancy:** The minimum of ${8, 32, 16}$ is **8 blocks per SM**.

#### III. Module 3 Lesson 3: Host Code

**3.1 Global Memory Management**

- **Visibility and Scope:** While global memory is visible to all CUDA blocks, it resides in **device memory** and is not directly accessible by the CPU due to the I/O interface.
- **Explicit Management APIs:**
    - `cudaMalloc`: Used to allocate memory space specifically within the GPU.
        - _Example:_ `cudaMalloc(&d_A, size);`
    - `cudaMemcpy`: Facilitates the transfer of data between the host (CPU) and the device (GPU).
        - **Directional Flags:**
            - `cudaMemcpyHostToDevice`: Transfer data from CPU to GPU.
            - `cudaMemcpyDeviceToHost`: Transfer data from GPU back to the CPU.
- **Unified Memory:** A programming feature (to be detailed in later modules) that eliminates the requirement for these explicit manual data copies.

**3.2 Full Vector Addition Architecture**

- **Host Code (CPU):** Responsible for setting up the environment, reading input files, allocating memory via `cudaMalloc`, and invoking the kernel.
- **Kernel Code (GPU):** Executed by individual threads in parallel to perform the actual mathematical operations (e.g., `c[idx] = a[idx] + b[idx]`).

#### IV. Module 3 Lesson 4: Stencil Operation with CUDA

**4.1 Stencil Operation Fundamentals**

- **Definition:** A common High-Performance Computing (HPC) operation where a value (e.g., "C") is computed by averaging its four immediate neighbors (East, West, North, South).
- **Programming Pattern:** Follows a **data decomposition** pattern where different CUDA blocks handle specific spatial "zones" (often visualized as different color zones).
- **Parallelism:** Each individual thread typically handles a single element of the stencil operation.

**4.2 Optimization via Shared Memory**

- **Reuse Factor:** Because each element is utilized at least four times by neighboring computations, it is a primary candidate for caching in shared memory to leverage on-chip speed.
- **Standard Three-Stage Pattern:**
    1. **Stage 1 (Load):** Threads load required data from global memory into the on-chip shared memory.
    2. **Stage 2 (Compute):** The processor performs the stencil calculation using the local shared memory data.
    3. **Stage 3 (Store):** The result is written back to global memory.
- **Synchronization Requirement:** A `__syncthreads()` call is essential between Stage 1 and Stage 2 to ensure all threads have finished loading the shared memory before any thread begins the computation stage.

**4.3 Boundary Handling and "Halo" Regions**

- **The Boundary Problem:** Since shared memory is local to a single block, calculating the value of an element at the edge of a block's zone requires data (neighboring elements) that may belong to a different block's zone.
- **Implementation Solution:**
    - Each block must explicitly bring "boundary" or "halo" elements into its own shared memory.
    - This is conceptually similar to MPI programming where explicit communication is required for boundary data.
    - **Thread Allocation:** The program must create a larger number of threads than there are inner elements. These additional threads are responsible for loading the boundary data from global memory to shared memory.

**4.4 Conditional Logic and Implementation Snippets**

- **Boundary Checking:** `if-else` statements are used within the kernel to determine if a thread's position (`idx`) requires it to load data from global memory or fill a boundary with a default value (e.g., zero).
- **Operational Pattern:**
    - **Load Phase:**
        
        ```
        // Logical representation of loading boundary values
        if (thread_is_on_boundary) {
            shared_mem[local_idx] = global_mem[global_idx];
        } else {
            shared_mem[local_idx] = 0; // Or other boundary condition
        }
        ```
        
    - **Computation Phase:**
        
        ```
        // Perform computations only on the designated inner elements
        if (thread_is_in_inner_zone) {
            // Stencil calculation logic
        }
        ```
        
- **Design Note:** Using a convolutional style for stencil operations can help reduce the number of explicit `if-else` statements required in the code.
