# **GPU Compiler Analysis and Optimization Manual: Part 1**

This technical manual provides a structural synthesis of GPU compiler backgrounds, focusing on advanced data-flow analysis, intermediate representations, and optimization techniques.

## **I. Live-Variable (Liveness) Analysis**

### **1.1 Concept and Application**

Liveness analysis is a critical technique used to identify which variables remain "in use" at specific program points during execution.

- **Primary Application:** **Register Allocation**. Registers are reserved exclusively for variables that are actively in use.
- **Efficiency:** By ensuring registers are only allocated to live variables, the compiler optimizes register spills, enhancing overall performance.
- **Reusability:** If a variable is no longer live after a specific block, its register can be reused for a different variable.

### **1.2 Data-Flow Equations**

Liveness analysis is a **backward analysis** process, moving from the program exit toward the entry. The analysis is defined by the following sets and equations:

- **$def_B$:** The set of variables defined in basic block $B$ before any use.
- **$use_B$:** The set of variables whose values may be used in block $B$ before any definition.
- **Boundary Condition ($IN[EXIT]$):** Specifies that no variables are alive upon exiting the program ($IN[EXIT] = 0$).
- **$IN[B]$ Equation:**
    - $IN[B] = use_B \cup (OUT[B] - def_B)$.
    - **Logic:** A variable is live entering a block if it is used before redefinition within that block, or if it is live exiting the block and was not redefined inside it.
- **$OUT[B]$ Equation:**
    - $OUT[B] = \bigcup_{S \in succ(B)} IN[S]$.
    - **Logic:** A variable is live exiting a block if and only if it is live entering at least one of its successor blocks.

### **1.3 Iterative Liveness Algorithm**

1. **Initialization:**
    - Set $IN[EXIT] = 0$.
    - Initialize all other $IN[B]$ values to $0$.
2. **Iteration:** Execute a loop through each basic block in backward order until the values stabilize (no changes occur).
    - Compute $OUT[B]$ by combining $IN$ values of all successors.
    - Apply the transfer function: $IN[B] = use_B \cup (OUT[B] - def_B)$.

### **1.4 Technical Case Study: Liveness Trace**

Based on a sample Control Flow Graph (CFG) with basic blocks B1, B2, B3, and B4:

#### **Initial Sets (Def/Use):**

- **B1:** Defines ${i, j, a}$; Uses ${m, n, u1}$.
- **B2:** Defines ${i, n, j}$; Uses ${i, n, j}$.
- **B3:** Defines ${a}$; Uses ${u2}$.
- **B4:** Defines ${i}$; Uses ${u3}$.

#### **Pass 1 (Backward Trace):**

- **B4:** $OUT[B4] = 0$. $IN[B4] = (0 - {i}) \cup {u3} = {u3}$.
- **B3:** $OUT[B3] = IN[B4] = {u3}$. $IN[B3] = ({u3} - {a}) \cup {u2} = {u2, u3}$.
- **B2:** Successors are B3, B4. $OUT[B2] = IN[B3] \cup IN[B4] = {u2, u3}$. $IN[B2] = ({u2, u3} - {i, j}) \cup {i, j} = {i, j, u2, u3}$.
- **B1:** Successor is B2. $OUT[B1] = {i, j, u2, u3}$. $IN[B1] = ({i, j, u2, u3} - {i, j, a}) \cup {m, n, u1} = {m, n, u1, u2, u3}$.
- **ENTRY:** $OUT[ENTRY] = IN[B1] = {m, n, u1, u2, u3}$.

#### **Pass 2 (Backward Trace):**

- **B4:** Successor is B2. $OUT[B4] = IN[B2] = {i, j, u2, u3}$. $IN[B4] = ({i, j, u2, u3} - {i}) \cup {u3} = {j, u2, u3}$.
- **B3:** Successor is B4. $OUT[B3] = IN[B4] = {j, u2, u3}$. $IN[B3] = ({j, u2, u3} - {a}) \cup {u2} = {j, u2, u3}$.
- **B2:** $OUT[B2] = IN[B3] \cup IN[B4] = {j, u2, u3}$. $IN[B2]$ remains ${j, u2, u3}$ after applying its def/use (which are identical).
- **Convergence:** Values stabilize after the **third pass** as results match the second pass.

### **1.5 Register Allocation Mechanics**

- **Resource Constraints:** Most often, there are fewer physical registers than live variables.
- **Spills and Fills:** When registers are exhausted, the compiler performs "spills" (moving a value from a register to the memory stack) and "fills" (restoring the value from the stack back to a register).
- **PTX Abstraction:** Parallel Thread Execution (PTX) assumes an **infinite number of registers** to simplify analysis. Consequently, explicit stack operations for spills/fills are **not shown** in PTX code.

## **VIII. Live-Variable (Liveness) Analysis**

### **8.1 Concept and Core Purpose**

Liveness analysis is a backward data-flow analysis technique used to determine which variables are "live"—meaning their current values are required for future use—at any specific point in a program.

- **Definition of Liveness:** A variable is live at a program point if there is a path from that point to a use of the variable that does not go through a new definition.
- **Primary Application: Register Allocation.**
    - Registers are reserved only for live variables.
    - Once a variable is no longer live, its assigned register can be reused for another variable.
    - This optimization minimizes **register spills** (moving values to memory), thereby enhancing program performance.

### **8.2 Data-Flow Equations and Sets**

Liveness analysis utilizes the following sets and equations to track variable states:

- **$def_B$:** The set of variables defined (written to) in block $B$ before any use of that variable occurs in the same block.
- **$use_B$:** The set of variables whose values are used (read) in block $B$ before any definition of that variable occurs within the block.
- **Boundary Condition:** $IN[EXIT] = \emptyset$. No variables are considered live upon program exit.
- **$IN[B]$ (Transfer Function):**
    - $IN[B] = use_B \cup (OUT[B] - def_B)$.
    - **Logic:** A variable is live entering a block if it is either used before redefinition inside that block OR it was live exiting the block and was not redefined within it.
- **$OUT[B]$:**
    - $OUT[B] = \bigcup_{S \in succ(B)} IN[S]$.
    - **Logic:** A variable is live exiting a block if and only if it is live entering at least one of its successor blocks.

### **8.3 Iterative Liveness Algorithm**

1. **Initialize:** Set $IN[EXIT] = 0$ and all other $IN[B]$ values to $0$.
2. **Iterative Process:** Repeat the following until no values change:
    - For each basic block $B$ (processed in backward order):
        - Compute $OUT[B]$ by combining $IN$ values of all successors.
        - Apply the transfer function to compute $IN[B]$.

### **8.4 Technical Comparison: Reaching Definitions vs. Liveness**

|Feature|Reaching Definitions (RD)|Live-Variable Analysis (LV)|
|:--|:--|:--|
|**Domain**|Set of Definitions|Set of Variables|
|**Direction**|Forward (Entry to Exit)|Backward (Exit to Entry)|
|**Transfer Function**|$gen_B \cup (x - kill_B)$|$use_B \cup (x - def_B)$|
|**Boundary Condition**|Initialize $OUT[ENTRY]$|Initialize $IN[EXIT]$|
|**Meet Operation**|Union ($\cup$)|Union ($\cup$)|

### **8.5 Register Allocation Constraints in PTX**

While the compiler must manage physical resource constraints:

- **Spills and Fills:** When live variables exceed available registers, the compiler performs "spills" (storing values in the memory stack) and "fills" (restoring them).
- **PTX Abstraction:** **PTX assumes an infinite number of registers** to simplify the representation. Consequently, explicit stack operations for spills/fills are omitted in PTX code.

## **IX. Static Single-Assignment (SSA) Form**

### **9.1 Fundamentals of SSA**

Static Single-Assignment (SSA) is an intermediate representation (IR) form that optimizes the traditional def-use chain.

- **Core Characteristic:** Each variable is assigned a **unique definition point**; it can be defined exactly once within the program.
- **Versioning:** Every time a variable (e.g., `a`) is redefined, it receives a new version name (e.g., `a1`, `a2`, `a3`).
- **Benefits:** SSA simplifies analysis and optimization passes by eliminating the need to track multiple conflicting definitions for the same variable name.

### **9.2 $\Phi$ (Phi) Functions**

The $\Phi$ function is used at joint points in the Control Flow Graph (CFG) to merge values from different execution paths.

- **Usage:** Inserted when a block has more than one predecessor.
- **Logic:** It selects the "latest version" of a variable based on which path the program took to reach the joint point.
- **Implementation:** At the ISA level, $\Phi$ functions can be implemented via `move` instructions, or specialized methods like `select` or `conditional move`.

### **9.3 Path-Convergence Criterion**

To avoid generating excessive $\Phi$ functions, they are only inserted when multiple values exist on converging paths. A $\Phi$ function for variable `a` is required at node `z` if:

1. There is a block `x` containing a definition of `a`.
2. There is a separate block `y` (where $y \neq x$) also containing a definition of `a`.
3. There is a non-empty path from `x` to `z` and a non-empty path from `y` to `z`.
4. These paths from `x` and `y` have no nodes in common other than `z`.

### **9.4 SSA in LLVM**

- LLVM IR treats $\Phi$ functions as fundamental definitions.
- The literal keyword `phi` is used in LLVM code to represent these joins.
- Backends eventually convert these back to standard ISA-specific branches or conditional moves.

# **GPU Architecture and Compilation Manual: Part 5**

This final section of the manual details specific compiler optimization techniques and the principles of divergence analysis for GPU architectures.

## **X. Specific Compiler Optimization Techniques**

### **10.1 Loop Unrolling**

Loop unrolling is a technique where a loop is expanded, either manually or automatically, to execute multiple iterations within a single loop body.

- **Key Benefits:**
    - Improved instruction scheduling.
    - Increased opportunities for vectorization.
    - Reduction in the total number of control flow branches.
- **Mechanism:** Instead of incrementing the loop by a single iteration, it is unrolled by a factor (e.g., a factor of four).
- **Epilog Loop:** If the total number of iterations is not a multiple of the unrolling factor, an epilog loop is utilized to handle the remaining iterations.

### **10.2 Function Inlining**

Function inlining is a global (inter-procedural) optimization that replaces a function call with the actual code of the function.

- **Advantages:**
    - Eliminates the overhead associated with stack operations.
    - Facilitates further local optimizations by exposing the function's code to the caller's context.
- **Disadvantages:** It can significantly increase code size due to duplicated logic.
- **Example:** Replacing calls to `func1(aa)` and `func2(bb)` directly with the expressions `aa * ...` and `bb * ...` within the calling function.

### **10.3 Dead Code Elimination**

This process identifies and removes instructions that do not contribute to the final output of a program.

- **Impact:** Reduces code size and potentially improves execution speed.
- **PTX Example:**
    - **Original:** `mul.f32 c, a, b;`
    - **Logic:** If the variable `c` is never subsequently used or read, the instruction is identified as dead code and removed.

### **10.4 Constant Propagation**

Constant propagation replaces variables with their known constant values at compile time.

- **Optimization Cascade:** Replacing variables with constants often triggers further optimizations, such as constant folding.
- **PTX Example:**
    - If `a = 5.0` and `b = 3.0`, the instruction `mul.f32 c, a, b;` can be simplified to `mul.f32 c, 5.0, 3.0;`, which can further be reduced to the constant `15.0` and propagated throughout the program.

### **10.5 Strength Reduction**

Strength reduction involves replacing computationally expensive operations with cheaper, more efficient ones.

- **Examples:**
    - Replacing a **division by 2** with a **right shift** operation.
    - Replacing a **square operation** with a simple **multiplication**.

### **10.6 Code Motion: Loop-Invariant Instructions**

Code motion involves relocating instructions that do not change across loop iterations to a point outside the loop.

- **Loop Induction Variables:** Variables like the loop counter (e.g., PTX: `add.u32 r1, r1, 1`) are used to control iterations and are often the targets for identifying loop-invariant code.
- **Example:** Moving an assignment like `cons = 3.14` outside of a loop so it executes once rather than every iteration.

## **XI. Divergence Analysis**

### **11.1 Concept of Divergent Branches**

Divergence occurs when threads within a single warp execute different paths of a program.

- **Static Analysis Goal:** Detect at compile-time which branches might cause threads to diverge, requiring hardware support via the SIMT stack.
- **Reconvergence Points:** The compiler must identify and insert **immediate post-dominators** (reconvergence points) to signal where divergent threads should synchronize.

### **11.2 Vectorization and Lock-Step Execution**

Divergence analysis is rooted in traditional vectorization techniques.

- **Vectorization Test:** The compiler checks if all instructions within a loop or branch will execute in "lock-step".
- **GPU Inheritance:** GPU compilers use these evolved vectorization tests to determine if instructions within a branch can be executed uniformly across a warp.

### **11.3 Thread Dependency Identification**

A branch is considered divergent if its condition is dependent on **thread IDs**.

- **Formal Definition:** At program point $p$, a condition is divergent if it evaluates to different values for different threads.
- **Dependency Chain:**
    - **Direct:** `if (threadIdx.x < 10)`.
    - **Indirect:** `c = threadIdx.x; if (c < 10)`.
- **Memory Dependency:** If a branch depends on a memory location, it is only divergent if the value at that memory location varies by thread ID.

### **11.4 Analysis Mechanism**

Compilers check for thread dependency by constructing a **def-use chain of thread IDs**.

- **Iterative Searching:** The compiler uses an iterative process to see if thread ID definitions can reach a branch condition.
- **Conservatism vs. Precision:** Very conservative analysis might mark all loops as divergent. Effective algorithms aim to "trim down" non-divergent conditions to improve optimization.
